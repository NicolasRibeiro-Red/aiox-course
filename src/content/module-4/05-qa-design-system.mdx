# QA Loops e Design System Workflows

O AIOX vai além de um único review — ele implementa **loops automatizados de QA** que repetem o ciclo `Review → Fix → Re-review` até atingir qualidade aceitável ou escalar para intervenção humana. Este capítulo cobre o **QA Loop Orchestrator** e como ele se integra com o workflow de Design System.

---

## QA Loop Orchestrator

O QA Loop é um workflow automatizado que orquestra o ciclo completo de revisão iterativa. Ele executa até um máximo de iterações configurável (padrão: 5), rastreando os resultados de cada ciclo.

### Anatomia de uma Iteração

Cada iteração do loop segue 4 passos:

| Passo | Agente | Ação | Artefato |
|---|---|---|---|
| 1. Review | `@qa` (Quinn) | Executa `qa-review-story.md` com CodeRabbit | `gate-file.yaml` |
| 2. Verdict | Sistema | Avalia: APPROVE, REJECT ou BLOCKED | — |
| 3. Fix Request | `@qa` (Quinn) | Gera documento priorizado de correções | `fix-request.md` |
| 4. Apply Fixes | `@dev` (Dex) | Aplica correções e valida com testes | `fixes-applied.json` |

Após o passo 4, o loop incrementa a iteração e volta ao passo 1. O ciclo para quando:

- **APPROVE** — Story aprovada, sai do loop
- **BLOCKED** — Issue grave, escala imediatamente
- **Max iterações atingidas** — Escala para humano

---

### Status do Loop

O QA Loop mantém um arquivo de estado persistente (`qa/loop-status.json`) com o histórico completo:

```json
{
  "storyId": "STORY-42",
  "currentIteration": 3,
  "maxIterations": 5,
  "status": "in_progress",
  "history": [
    { "iteration": 1, "verdict": "REJECT", "issuesFound": 7, "issuesFixed": 5 },
    { "iteration": 2, "verdict": "REJECT", "issuesFound": 4, "issuesFixed": 3 },
    { "iteration": 3, "verdict": "APPROVE", "issuesFound": 1, "issuesFixed": null }
  ]
}
```

Os status possíveis são: `pending`, `in_progress`, `completed`, `stopped`, `escalated`.

---

### Comandos de Controle

| Comando | Ação |
|---|---|
| `*qa-loop {storyId}` | Inicia loop completo |
| `*qa-loop-review` | Inicia apenas do step review |
| `*qa-loop-fix` | Inicia apenas do step fix |
| `*stop-qa-loop` | Para loop e salva estado |
| `*resume-qa-loop` | Retoma loop parado/escalado |
| `*escalate-qa-loop` | Força escalação manual |
| `*qa-loop --reset` | Deleta status e reinicia |

<Callout type="tip" title="Stop e Resume">
Você pode parar o loop a qualquer momento com `*stop-qa-loop` e retomar depois com `*resume-qa-loop`. O estado é preservado incluindo iteração atual e histórico completo.
</Callout>

---

### Escalação Automática

O loop escala automaticamente em 4 cenários:

| Trigger | Razão | Ação |
|---|---|---|
| Max iterations | Loop atingiu o limite sem APPROVE | Escala com contexto completo |
| BLOCKED | QA retornou verdict BLOCKED | Escala imediatamente |
| Fix failure | @dev não conseguiu corrigir após retries | Escala com log de erros |
| Manual | Usuário executou `*escalate-qa-loop` | Escala sob demanda |

Quando escalado, o sistema prepara um pacote de contexto com todos os gate files, fix requests e um sumário de todas as iterações.

## O Fix Request — Comunicação Estruturada

Um dos artefatos mais importantes do QA Loop é o `fix-request.md`. Ele é um documento estruturado que o `@qa` gera para o `@dev`, contendo:

```markdown
# QA Fix Request: STORY-42

## Instructions for @dev
- Fix ONLY the issues listed below
- Do not add features or refactor unrelated code

## Summary
| Severity | Count | Status                    |
|----------|-------|---------------------------|
| CRITICAL | 2     | Must fix before merge     |
| MAJOR    | 3     | Should fix before merge   |
| MINOR    | 1     | Optional improvements     |

## Issues to Fix
### 1. [CRITICAL] SQL Injection in login handler
- Location: `src/auth/login.ts:42`
- Problem: Input not sanitized before query
- Verification: [ ] Run security tests
```

<Callout type="warning" title="Scope discipline">
O fix request inclui a instrução explícita: "Fix ONLY the issues listed below. Do not add features or refactor unrelated code." Isso evita scope creep durante correções.
</Callout>

## Design System Build Quality Workflow

O AIOX também oferece um workflow específico para construção e validação de design systems. Ele foca em:

### Checks de Qualidade para Design Systems

| Área | O que verifica |
|---|---|
| **Tokens** | Cores, tipografia, espaçamento consistentes |
| **Componentes** | Atomic Design (atoms → molecules → organisms) |
| **Acessibilidade** | Contraste WCAG 2.1, aria labels, teclado |
| **Responsividade** | Breakpoints cobertos, mobile-first |
| **Estados** | Loading, error, empty, hover, focus, disabled |
| **Documentação** | Props documentadas, exemplos de uso |

### Integração com QA Loop

O workflow de Design System se integra com o QA Loop de forma natural:

1. `@ux-design-expert` especifica os componentes e tokens
2. `@dev` implementa seguindo a especificação
3. `@qa` revisa focando em consistência visual, acessibilidade e estados
4. O QA Loop garante iterações até atingir os padrões

## Configuração do QA Loop

Os parâmetros do loop são configuráveis por projeto:

```yaml
# .aiox-core/core-config.yaml
autoClaude:
  qaLoop:
    maxIterations: 5        # Máximo de iterações
    reviewTimeout: 1800000  # 30 min para review
    fixTimeout: 3600000     # 60 min para fixes
```

Para projetos menores, você pode reduzir o número de iterações:

```yaml
autoClaude:
  qaLoop:
    maxIterations: 3
    reviewTimeout: 900000   # 15 min
    fixTimeout: 1800000     # 30 min
```

---

## Integração CodeRabbit no QA Loop

O `@qa` integra o CodeRabbit como parte do review, com self-healing automático:

```yaml
self_healing:
  enabled: true
  max_iterations: 3
  severity_filter: [CRITICAL, HIGH]
  behavior:
    CRITICAL: auto_fix
    HIGH: auto_fix
    MEDIUM: document_as_debt
    LOW: ignore
```

Isso significa que antes mesmo de gerar o verdict, o CodeRabbit tenta resolver issues CRITICAL e HIGH automaticamente.
