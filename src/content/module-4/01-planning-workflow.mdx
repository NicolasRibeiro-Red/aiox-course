# O Spec Pipeline — De Ideias a Especificações

O **Spec Pipeline** é o workflow que transforma requisitos informais em especificações formais e executáveis. Ele representa um dos pilares do AIOX: a ideia de que **antes de escrever código, agentes especializados devem planejar, pesquisar e validar** tudo o que será construído.

---

## Por que um Pipeline de Especificação?

Em projetos tradicionais, a especificação é muitas vezes informal — um Slack, um rascunho, uma conversa. Isso gera:

- Ambiguidade nos requisitos
- Retrabalho por falta de pesquisa prévia
- Falta de rastreabilidade entre requisito e código

O Spec Pipeline resolve isso com **6 fases estruturadas**, cada uma executada por um agente especializado, com quality gates obrigatórios entre elas.

## Adaptação por Complexidade

O pipeline não é rígido — ele se adapta automaticamente baseado na complexidade detectada na Fase 2. Isso evita overhead desnecessário para tarefas simples e garante profundidade para tarefas complexas.

| Classificação | Score | Fases Ativadas | Tempo Estimado |
|---|---|---|---|
| **SIMPLE** | 8 ou menos | Gather, Spec, Critique | 30-60 min |
| **STANDARD** | 9-15 | Todas as 6 fases | 2-4 horas |
| **COMPLEX** | 16 ou mais | Todas + Revise + 2a Critique | 4-8 horas |

<Callout type="info" title="Fluxo adaptativo">
Para projetos SIMPLE, o pipeline pula Assess, Research e Plan — indo direto de Gather para Spec e Critique. Isso economiza tempo sem sacrificar qualidade para tarefas triviais.
</Callout>

## O Princípio "No Invention"

Um dos princípios mais importantes do Spec Pipeline é o **Article IV — No Invention**. Ele determina que:

- Todo statement na especificação **deve rastrear** para um requisito funcional (`FR-*`), não-funcional (`NFR-*`), restrição (`CON-*`) ou achado de pesquisa verificado
- É **proibido** adicionar features não listadas, assumir detalhes não pesquisados, ou criar critérios de aceite inventados
- Informação sem rastreabilidade deve ir para a seção "Open Questions"

```
Exemplo de violação:
  "O sistema deve suportar cache Redis para melhor performance"
  ↳ Nenhum requisito menciona Redis. Nenhuma pesquisa validou.
  ↳ AÇÃO: Remover ou mover para Open Questions
```

## Os 4 Agentes do Pipeline

| Agente | Nome | Arquétipo | Fases |
|---|---|---|---|
| `@pm` | Morgan | Strategist | 1 (Gather), 4 (Spec), 5b (Revise) |
| `@architect` | Aria | Visionary | 2 (Assess), 6 (Plan) |
| `@analyst` | Atlas | Decoder | 3 (Research) |
| `@qa` | Quinn | Guardian | 5 (Critique), 5c (2a Critique) |

Cada agente traz uma perspectiva única. Morgan foca no **usuário e produto**, Aria na **arquitetura e viabilidade**, Atlas na **pesquisa e evidência**, e Quinn na **qualidade e rastreabilidade**.

---

## A Critique: Quality Gate Central

A Fase 5 é o ponto mais crítico do pipeline. Quinn avalia a especificação em 5 dimensões, cada uma com peso diferente:

| Dimensão | Peso | O que verifica |
|---|---|---|
| Accuracy | 25% | Spec reflete requisitos corretamente? |
| Completeness | 25% | Todas as seções preenchidas? Testes cobrem FRs? |
| Consistency | 20% | IDs válidos? Sem contradições? |
| Feasibility | 15% | Tecnicamente possível? Dependências existem? |
| Alignment | 15% | Alinhado com stack e padrões do projeto? |

O verdict pode ser:

- **APPROVED** — Sem issues HIGH, média >= 4.0, todas dimensões >= 3. Avança para Plan.
- **NEEDS_REVISION** — Issues MEDIUM ou média entre 3.0-3.9. Retorna para o PM revisar.
- **BLOCKED** — Issues HIGH ou média < 3.0. Escala para o Architect.

## Artefatos Gerados

Ao final do pipeline, 6 artefatos ficam disponíveis em `docs/stories/{storyId}/spec/`:

```
docs/stories/STORY-42/
├── spec/
│   ├── requirements.json    # Requisitos estruturados (9 categorias)
│   ├── complexity.json      # Avaliação de complexidade (5 dimensões)
│   ├── research.json        # Dependências pesquisadas e validadas
│   ├── spec.md              # Especificação completa executável
│   └── critique.json        # Resultado da avaliação de qualidade
└── plan/
    └── implementation.yaml  # Plano com subtasks atômicas
```

---

## Como Executar

O pipeline pode ser iniciado de duas formas:

```bash
# Comando direto
*create-spec STORY-42

# Ou automaticamente quando autoSpec está habilitado
# e uma story é criada
```

Para executar fases individuais:

```bash
*gather-requirements STORY-42   # Apenas Fase 1
*assess-complexity STORY-42     # Apenas Fase 2
*critique-spec STORY-42         # Apenas Fase 5
```

<Callout type="tip" title="Retomando execuções">
O pipeline suporta checkpoints. Se uma fase falhar, você pode retomar de onde parou com `*create-spec STORY-42 --resume`.
</Callout>
