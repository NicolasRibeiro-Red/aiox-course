# Story Development Cycle — O Loop Central

O **Story Development Cycle** e o workflow mais usado do AIOX. Ele automatiza o fluxo completo de uma story — desde a criacao ate a entrega com quality gate — seguindo a sequencia: **criar → validar → implementar → QA review**.

## Fluxo de Status da Story

Cada story passa por um ciclo de vida definido:

```
Draft → Ready → In Progress → In Review → Done
  ↑       ↑                       |
  |       |                       ↓
  └── PO rejeita          QA rejeita → volta para In Progress
```

| Status | Quem define | Proximo passo |
|---|---|---|
| **Draft** | @sm cria a story | @po valida |
| **Ready** | @po aprova (10 checks) | @dev implementa |
| **In Progress** | @dev inicia trabalho | @dev conclui |
| **In Review** | @dev termina + CodeRabbit | @qa revisa |
| **Done** | @qa aprova | Proxima story |

---

## Fase 2: Validacao — Os 10 Checks do PO

O `@po` (Pax) executa um checklist rigido antes de liberar para desenvolvimento:

| # | Check | O que verifica |
|---|---|---|
| 1 | Titulo claro | Descreve precisamente o que sera feito |
| 2 | Descricao completa | Problema/necessidade claramente explicado |
| 3 | Acceptance criteria testaveis | Formato Given/When/Then preferido |
| 4 | Escopo bem definido | O que esta IN e OUT claramente listado |
| 5 | Dependencias mapeadas | Stories ou recursos prerequisitos identificados |
| 6 | Estimativa de complexidade | Pontos ou T-shirt sizing |
| 7 | Valor de negocio | Beneficio para usuario/negocio claro |
| 8 | Riscos documentados | Potenciais problemas identificados |
| 9 | Criterios de Done | Definicao clara de quando esta completa |
| 10 | Alinhamento com PRD/Epic | Consistencia com documentos fonte |

Se qualquer check critico falhar, a story retorna para o `@sm` com feedback detalhado.

---

## Fase 3: Implementacao — 3 Modos de Execucao

O `@dev` (Dex) pode operar em tres modos diferentes, dependendo da complexidade e preferencia:

| Modo | Prompts | Quando usar |
|---|---|---|
| **YOLO** | 0-1 | Tarefas simples e deterministicas. Decisoes autonomas com logging. |
| **Interactive** (default) | 5-10 | Aprendizado e decisoes complexas. Checkpoints de feedback. |
| **Pre-Flight** | 10-15 | Requisitos ambiguos, trabalho critico. Planejamento completo antes de executar. |

### CodeRabbit Self-Healing Loop

Antes de marcar a story como "In Review", o @dev executa automaticamente o CodeRabbit:

```
1. Executar CodeRabbit scan
2. Se sem CRITICAL issues → documentar HIGH e prosseguir
3. Se CRITICAL encontrados → tentar auto-fix
4. Incrementar iteracao (max: 2)
5. Se issues persistem apos 2 tentativas → HALT
```

<Callout type="info" title="Self-Healing automatico">
O CodeRabbit Self-Healing tenta corrigir issues CRITICAL automaticamente ate 2 vezes. Issues MEDIUM sao documentados como tech debt. Issues LOW sao ignorados.
</Callout>

---

## Fase 4: QA Review — O Quality Gate

O `@qa` (Quinn) executa 7 checks antes de aprovar:

| # | Check | Descricao |
|---|---|---|
| 1 | Code review | Padroes, legibilidade, manutenibilidade |
| 2 | Testes unitarios | Adequados e passando |
| 3 | Acceptance criteria | Todos atendidos |
| 4 | Sem regressoes | Funcionalidades existentes preservadas |
| 5 | Performance | Dentro dos limites aceitaveis |
| 6 | Seguranca | OWASP basics verificados |
| 7 | Documentacao | Atualizada se necessario |

### Decisoes do Gate

| Decisao | Criterio | Acao |
|---|---|---|
| **PASS** | Todos checks passaram, sem issues HIGH | Story aprovada → Done |
| **CONCERNS** | Issues nao-bloqueantes presentes | Aprovada com observacoes |
| **FAIL** | Issues HIGH/CRITICAL presentes | Retorna para @dev com fixes |
| **WAIVED** | Issues explicitamente aceitos | Aprovada com waiver documentado |

---

## Condicoes de Bloqueio

O workflow HALT e solicita intervencao humana quando:

1. **Dependencias nao aprovadas** — nova lib ou recurso requerido
2. **Ambiguidade nos requisitos** — story nao clara apos verificacao
3. **3 falhas consecutivas** — tentativas de implementacao ou correcao
4. **Configuracao faltando** — core-config.yaml ou templates ausentes
5. **Regressao** — testes existentes quebraram

---

## Os 4 Agentes do Ciclo

| Agente | Nome | Arquetipo | Papel | Comandos Chave |
|---|---|---|---|---|
| `@sm` | River | Facilitator | Criacao de stories | `*draft` |
| `@po` | Pax | Balancer | Validacao de stories | `*validate-story-draft` |
| `@dev` | Dex | Builder | Implementacao | `*develop`, `*apply-qa-fixes` |
| `@qa` | Quinn | Guardian | Quality gate | `*review`, `*gate` |

---

## Quando Usar e Quando NAO Usar

| Cenario | Usar SDC? | Alternativa |
|---|---|---|
| Qualquer story (greenfield ou brownfield) | Sim | — |
| Ciclo completo com validacao e QA | Sim | — |
| Hotfixes urgentes | Nao | Fluxo simplificado sem QA gate |
| Spikes/POCs exploratorios | Nao | Desenvolvimento ad-hoc |
| Tasks puramente tecnicas sem story | Nao | Tarefas diretas com @dev |
