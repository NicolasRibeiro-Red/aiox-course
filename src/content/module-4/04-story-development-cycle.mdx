# Story Development Cycle — O Loop Central

O **Story Development Cycle** é o workflow mais usado do AIOX. Ele automatiza o fluxo completo de uma story — desde a criação até a entrega com quality gate — seguindo a sequência: **criar → validar → implementar → QA review**.

## Fluxo de Status da Story

Cada story passa por um ciclo de vida definido:

```
Draft → Ready → In Progress → In Review → Done
  ↑       ↑                       |
  |       |                       ↓
  └── PO rejeita          QA rejeita → volta para In Progress
```

| Status | Quem define | Próximo passo |
|---|---|---|
| **Draft** | @sm cria a story | @po valida |
| **Ready** | @po aprova (10 checks) | @dev implementa |
| **In Progress** | @dev inicia trabalho | @dev conclui |
| **In Review** | @dev termina + CodeRabbit | @qa revisa |
| **Done** | @qa aprova | Próxima story |

---

## Fase 2: Validação — Os 10 Checks do PO

O `@po` (Pax) executa um checklist rígido antes de liberar para desenvolvimento:

| # | Check | O que verifica |
|---|---|---|
| 1 | Título claro | Descreve precisamente o que será feito |
| 2 | Descrição completa | Problema/necessidade claramente explicado |
| 3 | Acceptance criteria testáveis | Formato Given/When/Then preferido |
| 4 | Escopo bem definido | O que está IN e OUT claramente listado |
| 5 | Dependências mapeadas | Stories ou recursos prerequisitos identificados |
| 6 | Estimativa de complexidade | Pontos ou T-shirt sizing |
| 7 | Valor de negócio | Benefício para usuário/negócio claro |
| 8 | Riscos documentados | Potenciais problemas identificados |
| 9 | Critérios de Done | Definição clara de quando está completa |
| 10 | Alinhamento com PRD/Epic | Consistência com documentos fonte |

Se qualquer check crítico falhar, a story retorna para o `@sm` com feedback detalhado.

---

## Fase 3: Implementação — 3 Modos de Execução

O `@dev` (Dex) pode operar em três modos diferentes, dependendo da complexidade e preferência:

| Modo | Prompts | Quando usar |
|---|---|---|
| **YOLO** | 0-1 | Tarefas simples e determinísticas. Decisões autônomas com logging. |
| **Interactive** (default) | 5-10 | Aprendizado e decisões complexas. Checkpoints de feedback. |
| **Pre-Flight** | 10-15 | Requisitos ambíguos, trabalho crítico. Planejamento completo antes de executar. |

### CodeRabbit Self-Healing Loop

Antes de marcar a story como "In Review", o @dev executa automaticamente o CodeRabbit:

```
1. Executar CodeRabbit scan
2. Se sem CRITICAL issues → documentar HIGH e prosseguir
3. Se CRITICAL encontrados → tentar auto-fix
4. Incrementar iteração (max: 2)
5. Se issues persistem após 2 tentativas → HALT
```

<Callout type="info" title="Self-Healing automático">
O CodeRabbit Self-Healing tenta corrigir issues CRITICAL automaticamente até 2 vezes. Issues MEDIUM são documentados como tech debt. Issues LOW são ignorados.
</Callout>

---

## Fase 4: QA Review — O Quality Gate

O `@qa` (Quinn) executa 7 checks antes de aprovar:

| # | Check | Descrição |
|---|---|---|
| 1 | Code review | Padrões, legibilidade, manutenibilidade |
| 2 | Testes unitários | Adequados e passando |
| 3 | Acceptance criteria | Todos atendidos |
| 4 | Sem regressões | Funcionalidades existentes preservadas |
| 5 | Performance | Dentro dos limites aceitáveis |
| 6 | Segurança | OWASP basics verificados |
| 7 | Documentação | Atualizada se necessário |

### Decisões do Gate

| Decisão | Critério | Ação |
|---|---|---|
| **PASS** | Todos checks passaram, sem issues HIGH | Story aprovada → Done |
| **CONCERNS** | Issues não-bloqueantes presentes | Aprovada com observações |
| **FAIL** | Issues HIGH/CRITICAL presentes | Retorna para @dev com fixes |
| **WAIVED** | Issues explicitamente aceitos | Aprovada com waiver documentado |

---

## Condições de Bloqueio

O workflow HALT e solicita intervenção humana quando:

1. **Dependências não aprovadas** — nova lib ou recurso requerido
2. **Ambiguidade nos requisitos** — story não clara após verificação
3. **3 falhas consecutivas** — tentativas de implementação ou correção
4. **Configuração faltando** — core-config.yaml ou templates ausentes
5. **Regressão** — testes existentes quebraram

---

## Os 4 Agentes do Ciclo

| Agente | Nome | Arquétipo | Papel | Comandos Chave |
|---|---|---|---|---|
| `@sm` | River | Facilitator | Criação de stories | `*draft` |
| `@po` | Pax | Balancer | Validação de stories | `*validate-story-draft` |
| `@dev` | Dex | Builder | Implementação | `*develop`, `*apply-qa-fixes` |
| `@qa` | Quinn | Guardian | Quality gate | `*review`, `*gate` |

---

## Quando Usar e Quando NÃO Usar

| Cenário | Usar SDC? | Alternativa |
|---|---|---|
| Qualquer story (greenfield ou brownfield) | Sim | — |
| Ciclo completo com validação e QA | Sim | — |
| Hotfixes urgentes | Não | Fluxo simplificado sem QA gate |
| Spikes/POCs exploratórios | Não | Desenvolvimento ad-hoc |
| Tasks puramente técnicas sem story | Não | Tarefas diretas com @dev |

## Na Prática: Uma Story Real do AIOX

Para ilustrar como uma story se parece na vida real, veja abaixo um trecho da **Story 6.1.4** do próprio framework AIOX — uma story de complexidade alta que unifica o sistema de greetings dos agentes:

```markdown
# Story 6.1.4: Unified Greeting System Integration (v4 - Expanded)

**Story ID:** 6.1.4
**Epic:** Epic-6.1 - Agent Identity System
**Wave:** Wave 1 (Foundation)
**Status:** Ready to Start
**Priority:** Critical
**Owner:** Dev (Dex)
**Duration:** 2.5 days (20 hours)
**Investment:** $250.00

---

## Objective

Implement a unified greeting system that integrates all greeting
components (session context, project status, agent personalization,
user preferences) into a single, optimized, and maintainable solution.
This story unifies Stories 6.1.1 through 6.1.6 into a cohesive system.

## Story

**As a** AIOX framework user,
**I want** agents to use a unified greeting system that integrates
session context, project status, agent personalization, and user
preferences,
**So that** I have a consistent, fast, and contextually relevant
experience when activating any agent.

## Business Value

- Unifies all previous greeting-related work (Stories 6.1.1-6.1.6)
- Fixes integration issues preventing components from working together
- Provides optimal performance (<150ms with timeout protection)
- Maintains backward compatibility with fallbacks
- Simplifies architecture by consolidating duplicate scripts

## Task Breakdown

### Already Implemented (But Not Integrated):
- Story 6.1.1: Agent Persona Definitions (greeting_levels)
- Story 6.1.2.4: Project Status Context (60s cache)
- Story 6.1.2.5: Contextual Agent Load Integration
- Story 6.1.2.6: Framework Configuration System
- Story 6.1.4 (Original): Greeting Preferences
- Story 6.1.6: Output Formatter

### New Scope (This Story):
- Expand agent-config-loader.js to load complete agent definitions
- Create unified generate-greeting.js wrapper
- Modify greeting-builder.js to accept pre-loaded context
- Update all 11 agents to use unified system
- Implement session state updates after commands
- Consolidate duplicate scripts
- Fix integration issues identified in technical review
```

### O que Observar Nesta Story

| Elemento | O que Faz Bem |
|----------|---------------|
| **Formato "As a... I want... So that..."** | Conecta implementação técnica ao valor para o usuário |
| **Escopo expandido documentado** | Mostra o que já existe vs o que falta — evita retrabalho |
| **Investment estimado ($250)** | Facilita decisão de prioridade com custo explícito |
| **Task breakdown granular** | O @dev sabe exatamente quais arquivos tocar e por quê |
| **Business Value explícito** | Justifica por que essa story existe (unifica 6 stories anteriores) |

Esta story é um exemplo de complexidade **real** — ela não cria algo do zero, mas unifica trabalho anterior que nunca foi integrado. Esse tipo de story "de integração" é extremamente comum em projetos de verdade e exige um escopo muito bem definido para não virar bola de neve.

Este é um trecho de uma story real do repositório AIOX — a Story 6.1.4 que unificou o sistema de greetings dos agentes. Observe a profundidade: header com metadata, formato "As a... I want... So that...", business value, contexto técnico e task breakdown.

```markdown
# Story 6.1.4: Unified Greeting System Integration (v4 - Expanded)

**Story ID:** 6.1.4

**Wave:** Wave 1 (Foundation)
**Status:** Ready to Start
**Priority:** Critical
**Owner:** Dev (Dex)
**Duration:** 2.5 days (20 hours)
**Investment:** $250.00

---

## Objective

Implement a unified greeting system that integrates all greeting
components (session context, project status, agent personalization,
user preferences) into a single, optimized, and maintainable solution.

## Story

**As a** AIOX framework user,
**I want** agents to use a unified greeting system that integrates
session context, project status, agent personalization, and user
preferences,
**So that** I have a consistent, fast, and contextually relevant
experience when activating any agent.

## Business Value
- Unifies all previous greeting-related work (Stories 6.1.1-6.1.6)
- Fixes integration issues preventing components from working together
- Provides optimal performance (<150ms with timeout protection)
- Maintains backward compatibility with fallbacks

## Context: Integration of Previous Stories

Story 6.1.1: Agent Persona Definitions — Implemented, but agents don't use it
Story 6.1.2.4: Project Status Context — Implemented, but agents don't use it
Story 6.1.2.5: Contextual Agent Load — Implemented, but agents don't use it

Current Problem: Agents use inline logic in STEP 3 instead of the
developed scripts. Claude Code doesn't have access to conversationHistory,
session detection always returns "new", commands always show "full" visibility.

Solution: Unified greeting generator called via Node.js:
  1. Execute: node .aiox-core/scripts/generate-greeting.js {agent-id}
  2. Capture the complete output
  3. Display the greeting exactly as returned
```

<Callout type="info" title="O que isso mostra">
Uma story real do AIOX tem 1400+ linhas. Inclui objetivo, business value, contexto técnico completo, análise de problemas atuais, solução proposta, task breakdown com checkboxes, e acceptance criteria formais. Não é um ticket de 3 linhas — é uma especificação executável.
</Callout>
