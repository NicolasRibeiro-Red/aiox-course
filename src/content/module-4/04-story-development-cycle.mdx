# Story Development Cycle — O Loop Central

O **Story Development Cycle** e o workflow mais usado do AIOX. Ele automatiza o fluxo completo de uma story — desde a criacao ate a entrega com quality gate — seguindo a sequencia: **criar → validar → implementar → QA review**.

## Fluxo de Status da Story

Cada story passa por um ciclo de vida definido:

```
Draft → Ready → In Progress → In Review → Done
  ↑       ↑                       |
  |       |                       ↓
  └── PO rejeita          QA rejeita → volta para In Progress
```

| Status | Quem define | Proximo passo |
|---|---|---|
| **Draft** | @sm cria a story | @po valida |
| **Ready** | @po aprova (10 checks) | @dev implementa |
| **In Progress** | @dev inicia trabalho | @dev conclui |
| **In Review** | @dev termina + CodeRabbit | @qa revisa |
| **Done** | @qa aprova | Proxima story |

---

## Fase 2: Validacao — Os 10 Checks do PO

O `@po` (Pax) executa um checklist rigido antes de liberar para desenvolvimento:

| # | Check | O que verifica |
|---|---|---|
| 1 | Titulo claro | Descreve precisamente o que sera feito |
| 2 | Descricao completa | Problema/necessidade claramente explicado |
| 3 | Acceptance criteria testaveis | Formato Given/When/Then preferido |
| 4 | Escopo bem definido | O que esta IN e OUT claramente listado |
| 5 | Dependencias mapeadas | Stories ou recursos prerequisitos identificados |
| 6 | Estimativa de complexidade | Pontos ou T-shirt sizing |
| 7 | Valor de negocio | Beneficio para usuario/negocio claro |
| 8 | Riscos documentados | Potenciais problemas identificados |
| 9 | Criterios de Done | Definicao clara de quando esta completa |
| 10 | Alinhamento com PRD/Epic | Consistencia com documentos fonte |

Se qualquer check critico falhar, a story retorna para o `@sm` com feedback detalhado.

---

## Fase 3: Implementacao — 3 Modos de Execucao

O `@dev` (Dex) pode operar em tres modos diferentes, dependendo da complexidade e preferencia:

| Modo | Prompts | Quando usar |
|---|---|---|
| **YOLO** | 0-1 | Tarefas simples e deterministicas. Decisoes autonomas com logging. |
| **Interactive** (default) | 5-10 | Aprendizado e decisoes complexas. Checkpoints de feedback. |
| **Pre-Flight** | 10-15 | Requisitos ambiguos, trabalho critico. Planejamento completo antes de executar. |

### CodeRabbit Self-Healing Loop

Antes de marcar a story como "In Review", o @dev executa automaticamente o CodeRabbit:

```
1. Executar CodeRabbit scan
2. Se sem CRITICAL issues → documentar HIGH e prosseguir
3. Se CRITICAL encontrados → tentar auto-fix
4. Incrementar iteracao (max: 2)
5. Se issues persistem apos 2 tentativas → HALT
```

<Callout type="info" title="Self-Healing automatico">
O CodeRabbit Self-Healing tenta corrigir issues CRITICAL automaticamente ate 2 vezes. Issues MEDIUM sao documentados como tech debt. Issues LOW sao ignorados.
</Callout>

---

## Fase 4: QA Review — O Quality Gate

O `@qa` (Quinn) executa 7 checks antes de aprovar:

| # | Check | Descricao |
|---|---|---|
| 1 | Code review | Padroes, legibilidade, manutenibilidade |
| 2 | Testes unitarios | Adequados e passando |
| 3 | Acceptance criteria | Todos atendidos |
| 4 | Sem regressoes | Funcionalidades existentes preservadas |
| 5 | Performance | Dentro dos limites aceitaveis |
| 6 | Seguranca | OWASP basics verificados |
| 7 | Documentacao | Atualizada se necessario |

### Decisoes do Gate

| Decisao | Criterio | Acao |
|---|---|---|
| **PASS** | Todos checks passaram, sem issues HIGH | Story aprovada → Done |
| **CONCERNS** | Issues nao-bloqueantes presentes | Aprovada com observacoes |
| **FAIL** | Issues HIGH/CRITICAL presentes | Retorna para @dev com fixes |
| **WAIVED** | Issues explicitamente aceitos | Aprovada com waiver documentado |

---

## Condicoes de Bloqueio

O workflow HALT e solicita intervencao humana quando:

1. **Dependencias nao aprovadas** — nova lib ou recurso requerido
2. **Ambiguidade nos requisitos** — story nao clara apos verificacao
3. **3 falhas consecutivas** — tentativas de implementacao ou correcao
4. **Configuracao faltando** — core-config.yaml ou templates ausentes
5. **Regressao** — testes existentes quebraram

---

## Os 4 Agentes do Ciclo

| Agente | Nome | Arquetipo | Papel | Comandos Chave |
|---|---|---|---|---|
| `@sm` | River | Facilitator | Criacao de stories | `*draft` |
| `@po` | Pax | Balancer | Validacao de stories | `*validate-story-draft` |
| `@dev` | Dex | Builder | Implementacao | `*develop`, `*apply-qa-fixes` |
| `@qa` | Quinn | Guardian | Quality gate | `*review`, `*gate` |

---

## Quando Usar e Quando NAO Usar

| Cenario | Usar SDC? | Alternativa |
|---|---|---|
| Qualquer story (greenfield ou brownfield) | Sim | — |
| Ciclo completo com validacao e QA | Sim | — |
| Hotfixes urgentes | Nao | Fluxo simplificado sem QA gate |
| Spikes/POCs exploratorios | Nao | Desenvolvimento ad-hoc |
| Tasks puramente tecnicas sem story | Nao | Tarefas diretas com @dev |

## Na Pratica: Uma Story Real do AIOX

Para ilustrar como uma story se parece na vida real, veja abaixo um trecho da **Story 6.1.4** do proprio framework AIOX — uma story de complexidade alta que unifica o sistema de greetings dos agentes:

```markdown
# Story 6.1.4: Unified Greeting System Integration (v4 - Expanded)

**Story ID:** 6.1.4
**Epic:** Epic-6.1 - Agent Identity System
**Wave:** Wave 1 (Foundation)
**Status:** Ready to Start
**Priority:** Critical
**Owner:** Dev (Dex)
**Duration:** 2.5 days (20 hours)
**Investment:** $250.00

---

## Objective

Implement a unified greeting system that integrates all greeting
components (session context, project status, agent personalization,
user preferences) into a single, optimized, and maintainable solution.
This story unifies Stories 6.1.1 through 6.1.6 into a cohesive system.

## Story

**As a** AIOX framework user,
**I want** agents to use a unified greeting system that integrates
session context, project status, agent personalization, and user
preferences,
**So that** I have a consistent, fast, and contextually relevant
experience when activating any agent.

## Business Value

- Unifies all previous greeting-related work (Stories 6.1.1-6.1.6)
- Fixes integration issues preventing components from working together
- Provides optimal performance (<150ms with timeout protection)
- Maintains backward compatibility with fallbacks
- Simplifies architecture by consolidating duplicate scripts

## Task Breakdown

### Already Implemented (But Not Integrated):
- Story 6.1.1: Agent Persona Definitions (greeting_levels)
- Story 6.1.2.4: Project Status Context (60s cache)
- Story 6.1.2.5: Contextual Agent Load Integration
- Story 6.1.2.6: Framework Configuration System
- Story 6.1.4 (Original): Greeting Preferences
- Story 6.1.6: Output Formatter

### New Scope (This Story):
- Expand agent-config-loader.js to load complete agent definitions
- Create unified generate-greeting.js wrapper
- Modify greeting-builder.js to accept pre-loaded context
- Update all 11 agents to use unified system
- Implement session state updates after commands
- Consolidate duplicate scripts
- Fix integration issues identified in technical review
```

### O que Observar Nesta Story

| Elemento | O que Faz Bem |
|----------|---------------|
| **Formato "As a... I want... So that..."** | Conecta implementacao tecnica ao valor para o usuario |
| **Escopo expandido documentado** | Mostra o que ja existe vs o que falta — evita retrabalho |
| **Investment estimado ($250)** | Facilita decisao de prioridade com custo explicito |
| **Task breakdown granular** | O @dev sabe exatamente quais arquivos tocar e por que |
| **Business Value explicito** | Justifica por que essa story existe (unifica 6 stories anteriores) |

Esta story e um exemplo de complexidade **real** — ela nao cria algo do zero, mas unifica trabalho anterior que nunca foi integrado. Esse tipo de story "de integracao" e extremamente comum em projetos de verdade e exige um escopo muito bem definido para nao virar bola de neve.

Este e um trecho de uma story real do repositorio AIOX — a Story 6.1.4 que unificou o sistema de greetings dos agentes. Observe a profundidade: header com metadata, formato "As a... I want... So that...", business value, contexto tecnico e task breakdown.

```markdown
# Story 6.1.4: Unified Greeting System Integration (v4 - Expanded)

**Story ID:** 6.1.4
**Epic:** Epic-6.1 - Agent Identity System
**Wave:** Wave 1 (Foundation)
**Status:** Ready to Start
**Priority:** Critical
**Owner:** Dev (Dex)
**Duration:** 2.5 days (20 hours)
**Investment:** $250.00

---

## Objective

Implement a unified greeting system that integrates all greeting
components (session context, project status, agent personalization,
user preferences) into a single, optimized, and maintainable solution.

## Story

**As a** AIOX framework user,
**I want** agents to use a unified greeting system that integrates
session context, project status, agent personalization, and user
preferences,
**So that** I have a consistent, fast, and contextually relevant
experience when activating any agent.

## Business Value
- Unifies all previous greeting-related work (Stories 6.1.1-6.1.6)
- Fixes integration issues preventing components from working together
- Provides optimal performance (<150ms with timeout protection)
- Maintains backward compatibility with fallbacks

## Context: Integration of Previous Stories

Story 6.1.1: Agent Persona Definitions — Implemented, but agents don't use it
Story 6.1.2.4: Project Status Context — Implemented, but agents don't use it
Story 6.1.2.5: Contextual Agent Load — Implemented, but agents don't use it

Current Problem: Agents use inline logic in STEP 3 instead of the
developed scripts. Claude Code doesn't have access to conversationHistory,
session detection always returns "new", commands always show "full" visibility.

Solution: Unified greeting generator called via Node.js:
  1. Execute: node .aiox-core/scripts/generate-greeting.js {agent-id}
  2. Capture the complete output
  3. Display the greeting exactly as returned
```

Uma story real do AIOX tem 1400+ linhas. Inclui objetivo, business value, contexto tecnico completo, analise de problemas atuais, solucao proposta, task breakdown com checkboxes, e acceptance criteria formais. Nao e um ticket de 3 linhas — e uma especificacao executavel.
</Callout>
