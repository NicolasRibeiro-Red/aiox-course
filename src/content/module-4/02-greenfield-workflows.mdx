# Greenfield Workflows — Criando Projetos do Zero

O **Greenfield Fullstack Workflow** é o fluxo principal do AIOX para construir aplicações completas desde o conceito até a entrega. Ele orquestra 9 agentes em 4 fases progressivas — do bootstrap do ambiente até o ciclo de desenvolvimento iterativo.

---

## Quando Usar Greenfield

Use workflows greenfield quando:

- Você está construindo uma aplicação nova, do zero
- O projeto precisa de documentação abrangente (PRD, arquitetura, specs)
- Há expectativa de manutenção de longo prazo
- Múltiplos membros vão colaborar no projeto

| Tipo de Projeto | Suportado |
|---|---|
| Web App | Sim |
| SaaS | Sim |
| Enterprise App | Sim |
| Prototype / POC | Sim |
| MVP | Sim |

## Fase 0: Bootstrap — Preparando o Terreno

Antes de qualquer planejamento, o `@devops` (Gage) garante que o ambiente está pronto. Ele verifica e instala automaticamente as ferramentas necessárias:

| Categoria | Ferramenta | Obrigatória |
|---|---|---|
| Essencial | git, gh (GitHub CLI), node, npm | Sim |
| Infraestrutura | supabase, docker | Recomendada |
| Qualidade | coderabbit | Recomendada |
| Opcional | pnpm, bun | Opcional |

<Callout type="tip" title="Pular o Bootstrap">
Se o projeto já tem um `.aiox/environment-report.json`, a Fase 0 é automaticamente pulada. Re-execute ao trocar de máquina ou quando novos membros entrarem.
</Callout>

## Fase 1: Descoberta — O Pipeline de Planejamento

A Fase 1 é um pipeline sequencial de 5 agentes, onde cada um consome o artefato do anterior:

```
@analyst → project-brief.md
    ↓
@pm → prd.md
    ↓
@ux-expert → front-end-spec.md
    ↓
@architect → fullstack-architecture.md
    ↓
@po → validação final (po-master-checklist)
```

### Pontos de Decisão na Fase 1

| Decisão | Opções | Critério |
|---|---|---|
| Gerar prompt v0? | Sim / Não | Usuário quer gerar UI com IA (v0, Lovable) |
| Arquitetura sugere mudanças no PRD? | Atualizar / Continuar | Recomendação do @architect |
| PO encontrou problemas? | Corrigir / Aprovar | Resultado do checklist |

Se o `@architect` detecta que a arquitetura exige mudanças nos requisitos, o PRD é atualizado pelo `@pm` antes de prosseguir. Se o `@po` encontra problemas, o artefato retorna ao agente responsável.

## Fase 2: Fragmentação — Documentos Prontos para Dev

O `@po` pega o PRD monolítico e o fragmenta em partes menores usando a task `shard-doc`:

```
docs/prd/
├── index.md          # Índice com links
├── overview.md       # Seção 1
├── features.md       # Seção 2
├── requirements.md   # Seção 3
└── ...

docs/architecture/
├── source-tree.md    # Estrutura de diretórios
├── tech-stack.md     # Stack tecnológica
└── coding-standards.md  # Padrões de código
```

Essa fragmentação permite que agentes na Fase 3 trabalhem com documentos focados em vez de ler o PRD inteiro.

---

## Fase 3: Desenvolvimento Iterativo

O ciclo de desenvolvimento segue um loop:

1. **@sm cria** a próxima story baseada nos documentos fragmentados
2. **Revisão opcional** — @analyst ou @pm revisam se a story está clara
3. **@dev implementa** — código, testes, CodeRabbit self-healing
4. **@qa revisa** — quality gate com checklist de 7 pontos
5. **@dev corrige** — se QA encontrar problemas
6. **Repete** — até completar todas as stories do epic

O status de cada story segue o fluxo: `Draft → Approved → In Progress → Review → Done`.

## Handoff entre Fases

O AIOX fornece **prompts de handoff** padronizados para transições entre agentes e fases. Isso garante que o contexto não se perde:

| De | Para | Handoff |
|---|---|---|
| Fase 0 → 1 | @devops → @analyst | "Bootstrap completo. Inicie com @analyst para criar o project brief." |
| @analyst → @pm | Atlas → Morgan | "Brief completo. Salve como `docs/project-brief.md`, depois crie o PRD." |
| @pm → @ux | Morgan → Uma | "PRD pronto. Salve como `docs/prd.md`, depois crie a spec de UI/UX." |
| Fase 1 → 2 | @po | "Artefatos validados. Fragmente documentos: @po *shard-doc docs/prd.md" |
| Fase 2 → 3 | @sm | "Documentos fragmentados. Inicie desenvolvimento: @sm *draft" |

<Callout type="warning" title="Contexto entre sessões">
Se você precisa trocar de sessão entre fases, sempre salve os artefatos gerados em disco antes. O handoff prompt só funciona se os arquivos existirem.
</Callout>

## Variantes de Greenfield

Além do fullstack, o AIOX oferece workflows greenfield especializados:

| Workflow | Uso | Diferença |
|---|---|---|
| `greenfield-fullstack` | Apps completas (frontend + backend + DB) | Pipeline completo com 9 agentes |
| `greenfield-service` | APIs e microsserviços | Sem @ux-expert, foco em arquitetura de APIs |
| `greenfield-ui` | Interfaces e SPAs | Sem @data-engineer, foco em componentes UI |

## Na Prática: Workflow YAML Real

Abaixo está o trecho real do arquivo `greenfield-fullstack.yaml` -- a definição completa do workflow que orquestra as 4 fases. Note como cada step especifica o agente responsável, os artefatos que cria, a duração estimada e as dependências:

```yaml
workflow:
  id: greenfield-fullstack
  name: Greenfield Full-Stack Application Development
  description: >-
    Agent workflow for building full-stack applications from concept to development.
    Supports both comprehensive planning for complex projects and rapid prototyping
    for simple ones. Includes Phase 0 environment bootstrap for proper tooling setup.
  type: greenfield
  version: "1.0"
  project_types:
    - web-app
    - saas
    - enterprise-app
    - prototype
    - mvp

  metadata:
    elicit: true
    confirmation_required: true

  phases:
    - phase_0: Environment Bootstrap
    - phase_1: Discovery & Planning
    - phase_2: Document Sharding
    - phase_3: Development Cycle

  sequence:
    # PHASE 0: ENVIRONMENT BOOTSTRAP
    - phase: 0
      name: Environment Bootstrap
      description: "Setup development environment before starting project planning"

    - id: devops-environment-bootstrap
      agent: devops
      action: environment_bootstrap
      duration_estimate: "15-30 min"
      creates:
        - .aiox/config.yaml
        - .aiox/environment-report.json
        - .gitignore
        - README.md
        - package.json
      task: environment-bootstrap.md
      optional: false
      notes: |
        CRITICAL FIRST STEP - Run before any planning:
        - @devops → *environment-bootstrap
        - Verifies/installs CLIs (git, gh, node, supabase, railway)
        - Authenticates services (GitHub, Supabase, Railway)
        - Creates Git repository locally and on GitHub
        - Scaffolds AIOX project structure

    # PHASE 1: DISCOVERY & PLANNING
    - phase: 1
      name: Discovery & Planning
      description: "Create all planning artifacts"

    - id: analyst-project-brief
      agent: analyst
      creates: project-brief.md
      duration_estimate: "30-60 min"
      requires: environment_bootstrap_complete
      optional_steps:
        - brainstorming-session
        - market-research-prompt
      notes: "Can do brainstorming first, then optional deep research
              before creating project brief."

    - id: pm-create-prd
      agent: pm
      creates: prd.md
      duration_estimate: "30-60 min"
```

<Callout type="info" title="Workflow como contrato">
O campo `creates` em cada step funciona como um contrato -- define exatamente quais artefatos aquele agente deve produzir. O campo `requires` estabelece pré-condições, garantindo que a sequência de fases seja respeitada.
</Callout>
