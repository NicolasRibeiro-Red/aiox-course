# Auto Worktree e Git Workflow

O AIOX implementa duas estratégias complementares para gerenciar código: **Auto Worktree** para isolamento de desenvolvimento e **Defense in Depth** para validação em múltiplas camadas. Juntas, elas garantem que stories sejam desenvolvidas em paralelo sem conflitos e que código ruim nunca chegue à branch principal.

---

## Auto Worktree — Isolamento por Story

O Auto Worktree é um workflow que cria **worktrees Git isoladas** para cada story. Cada story trabalha em um diretório e branch separados, permitindo desenvolvimento paralelo sem interferência.

### Como Funciona

Quando o `@dev` inicia uma story, o workflow automaticamente:

1. **Extrai o Story ID** do contexto (parâmetro, arquivo, task ou branch)
2. **Verifica se já existe** uma worktree para essa story
3. **Limpa worktrees obsoletas** (> 30 dias sem uso, se habilitado)
4. **Cria a worktree** em `.aiox/worktrees/{storyId}/` com branch `auto-claude/{storyId}`
5. **Troca o contexto** para o novo diretório
6. **Exibe o resumo** com instruções

### Estrutura Criada

```
.aiox/worktrees/
├── STORY-42/          # Worktree da story 42
│   ├── src/           # Cópia isolada do código
│   └── ...
├── STORY-43/          # Worktree da story 43
│   └── ...

Branches:
├── main
├── auto-claude/STORY-42
└── auto-claude/STORY-43
```

### Comandos de Worktree

| Comando | Descrição |
|---|---|
| `*create-worktree {storyId}` | Criar worktree manualmente |
| `*list-worktrees` | Listar worktrees ativas |
| `*remove-worktree {storyId}` | Remover worktree |
| `*merge-worktree {storyId}` | Fazer merge da worktree |
| `*cleanup-worktrees` | Limpar worktrees obsoletas |

<Callout type="info" title="Requisitos">
O Auto Worktree requer Git >= 2.5. Verifique com `git --version`. O limite padrão é 10 worktrees simultâneas, configurável em `.aiox/config.yaml`.
</Callout>

## Defense in Depth — Validação em 3 Camadas

O AIOX implementa uma estratégia de **defesa em profundidade** com 3 camadas de validação progressiva. Cada camada captura tipos diferentes de problemas no momento mais adequado.

### As 3 Camadas

| Camada | Tipo | Velocidade | Trigger | O que captura |
|---|---|---|---|---|
| **Layer 1** | Local | < 5s | `git commit` | Syntax, linting, type errors |
| **Layer 2** | Local | < 2s | `git push` | Story validation, status consistency |
| **Layer 3** | Cloud | 2-5 min | PR / CI | Full test suite, coverage, branch protection |

### Layer 1: Pre-commit Hook

Executado automaticamente em cada `git commit`. Usa cache para velocidade:

```bash
# .husky/pre-commit
#!/bin/sh
npm run lint      # ESLint com cache (.eslintcache)
npm run typecheck # TypeScript incremental (.tsbuildinfo)
```

Performance: primeira execução ~10-15s, subsequentes < 5s graças ao cache.

### Layer 2: Pre-push Hook

Executado automaticamente em cada `git push`. Valida stories:

```bash
# .husky/pre-push
#!/bin/sh
node .aiox-core/utils/aiox-validator.js pre-push
```

Regras de validação:

| Regra | Descrição |
|---|---|
| Status Consistency | Story `completed` mas tasks `[ ]`? Erro. |
| Required Sections | id, title, description, acceptance_criteria, status |
| Checkbox Format | Apenas `[x]` ou `[ ]` reconhecidos |
| Status Flow | `ready → in progress → Ready for Review → completed` |

### Layer 3: CI/CD Pipeline (GitHub Actions)

Executado na nuvem quando código é enviado ao GitHub:

```yaml
# .github/workflows/ci.yml
jobs:
  lint:           # ESLint sem cache (clean run)
  typecheck:      # TypeScript completo
  test:           # Jest com coverage (>= 80%)
  story-validation: # Todas stories validadas
  validation-summary: # Agrega resultados
```

## Branch Protection

O AIOX configura proteção automática para a branch `master`:

| Proteção | Configuração |
|---|---|
| Status checks obrigatórios | ESLint, TypeScript, Tests, Story Validation |
| Reviews obrigatórias | 1 aprovação mínima |
| Dismiss stale reviews | Sim, em novos commits |
| Histórico linear | Sim (rebase only) |
| Force push | Bloqueado |
| Deleção da branch | Bloqueada |

```bash
# Configurar proteção automaticamente
node scripts/setup-branch-protection.js

# Verificar status
node scripts/setup-branch-protection.js --status
```

## Fluxo Diário Completo

Um dia típico de desenvolvimento com AIOX segue este fluxo:

```bash
# 1. Atualizar master
git checkout master && git pull

# 2. Criar branch (ou worktree)
git checkout -b feature/minha-feature
# ou: *create-worktree STORY-42

# 3. Desenvolver e commitar
# → Pre-commit hook roda lint + typecheck
git add . && git commit -m "feat: add feature [Story X.X]"

# 4. Push
# → Pre-push hook valida stories
git push origin feature/minha-feature

# 5. Criar PR
# → CI roda todos os jobs
gh pr create --title "feat: Add feature" --body "Description"

# 6. Review e merge
# → Branch protection exige aprovação
gh pr merge --squash --delete-branch
```

<Callout type="warning" title="Evite --no-verify">
Os comandos `git commit --no-verify` e `git push --no-verify` pulam os hooks de validação. Use apenas em emergências absolutas (hotfixes críticos). Validações puladas não capturam problemas.
</Callout>

## Resolução de Problemas Comuns

| Problema | Causa | Solução |
|---|---|---|
| Hook não executa | Husky não instalado | `npm run prepare` |
| Pre-commit lento (> 10s) | Cache inválido | `rm .eslintcache .tsbuildinfo` |
| Story validation falha | Checkbox inconsistente | Verificar `[x]` vs status |
| CI falha mas local passa | Cache local diverge | `rm -rf node_modules && npm ci` |
| Worktree não cria | Git < 2.5 | Atualizar Git |
| Limite de worktrees | >= 10 ativas | `*cleanup-worktrees` |
| Conflito no merge | Branches divergiram | `git rebase main` dentro da worktree |
