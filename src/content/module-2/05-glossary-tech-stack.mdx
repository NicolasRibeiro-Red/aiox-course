# Glossario e Tech Stack

Para usar o AIOX com fluencia, voce precisa dominar sua terminologia oficial e entender o stack tecnologico que sustenta o framework. Nesta licao, consolidamos os termos-chave e exploramos cada camada da stack.

---

## Glossario Oficial AIOX 4.x

### Termos Oficiais

| Termo | Definicao | Quando Usar |
|---|---|---|
| **Squad** | Grupo de agentes AI especializados para um dominio/workstream | Descrevendo bundles de dominio e conjuntos reutilizaveis de agentes |
| **Flow-State** | Estado de progressao de workflow determinado em runtime | Referindo-se a orquestracao baseada em estado |
| **Confidence Gate** | Gate de decisao baseado em score/threshold de confianca | Discutindo decisoes de merge/block baseadas em confidence score |
| **Execution Profile** | Perfil de autonomia baseado em risco (`safe`, `balanced`, `aggressive`) | Controlando autonomia de agentes por contexto/risco |

<Callout type="warning" title="Termos depreciados">
Os seguintes termos foram depreciados e nao devem ser usados em documentacao nova:
- **expansion pack** → Use **squad**
- **permission mode** → Use **execution profile**
- **workflow state** → Use **flow-state**
</Callout>

### Termos Fundamentais do Framework

| Termo | Definicao |
|---|---|
| **Agent** | Persona AI com expertise especifica, definida em Markdown + YAML |
| **Task** | Instrucao executavel passo-a-passo que agentes seguem |
| **Template** | Estrutura YAML para geracao de documentos com elicitacao |
| **Workflow** | Sequencia prescrita de steps e interacoes multi-agente |
| **Checklist** | Lista de validacao que funciona como quality gate |
| **Agent Team** | Colecao pre-configurada de agentes para um proposito especifico |
| **Elicitation** | Processo interativo de refinamento de conteudo com o usuario |
| **Story** | Unidade atomica de trabalho com acceptance criteria e tasks |
| **Quality Gate** | Ponto de validacao obrigatorio entre fases do workflow |
| **Greenfield** | Projeto completamente novo, sem codigo existente |
| **Brownfield** | Projeto existente que recebe melhorias ou refactoring |
| **Document Sharding** | Fragmentacao de documentos grandes em partes consomiveis |
| **Knowledge Base (KB)** | Base de conhecimento do framework em `aiox-kb.md` |
| **Technical Preferences** | Perfil de preferencias tecnicas que influencia recomendacoes |

### Semantic Lint Policy

O AIOX aplica **linting semantico** automatizado via `scripts/semantic-lint.js`:

- Termos depreciados em nivel **error** bloqueiam commits e CI
- Termos em nivel **warning** sao reportados para migracao gradual

Isso garante consistencia terminologica em toda a documentacao e codigo.

## Tech Stack

### Filosofia do Stack

A escolha tecnologica do AIOX segue dois principios:

1. **Boring technology onde possivel** -- dependencias provadas e estaveis
2. **Exciting technology onde necessario** -- apenas quando performance ou DX justificam

### Core Runtime

```yaml
Node.js: 18.0.0+ (Active LTS)
Package Manager: npm 9.0.0+
Module System: CommonJS (ESM migration planejada)
```

**Por que Node.js 18+?**
- Native `fetch()` API -- sem necessidade de axios
- ES2022 support completo (top-level await, class fields)
- Cross-platform (Windows, macOS, Linux)
- Active LTS com patches de seguranca

## Variaveis de Ambiente

```bash
# AIOX Configuration
AIOX_DEBUG=true                    # Habilita logs de debug
AIOX_CONFIG_PATH=/custom/path      # Caminho customizado para config
AIOX_YOLO_MODE=true               # Forca modo YOLO (sem confirmacoes)

# Node.js
NODE_ENV=production                # Modo de producao
NODE_OPTIONS=--max-old-space-size=4096  # Limite de memoria

# Integracoes
CLICKUP_API_KEY=pk_xxx            # ClickUp
GITHUB_TOKEN=ghp_xxx              # GitHub API
RAILWAY_TOKEN=xxx                 # Railway deploy
SUPABASE_ACCESS_TOKEN=xxx         # Supabase CLI
```

<Callout type="danger" title="Seguranca">
NUNCA comite variaveis de ambiente com tokens ou API keys. Use `.env` (gitignored) e verifique seu `.gitignore` antes de cada push.
</Callout>
