# Glossário e Tech Stack

Para usar o AIOX com fluência, você precisa dominar sua terminologia oficial e entender o stack tecnológico que sustenta o framework. Nesta lição, consolidamos os termos-chave e exploramos cada camada da stack.

---

## Glossário Oficial AIOX 4.x

### Termos Oficiais

| Termo | Definição | Quando Usar |
|---|---|---|
| **Squad** | Grupo de agentes AI especializados para um domínio/workstream | Descrevendo bundles de domínio e conjuntos reutilizáveis de agentes |
| **Flow-State** | Estado de progressão de workflow determinado em runtime | Referindo-se à orquestração baseada em estado |
| **Confidence Gate** | Gate de decisão baseado em score/threshold de confiança | Discutindo decisões de merge/block baseadas em confidence score |
| **Execution Profile** | Perfil de autonomia baseado em risco (`safe`, `balanced`, `aggressive`) | Controlando autonomia de agentes por contexto/risco |

<Callout type="warning" title="Termos depreciados">
Os seguintes termos foram depreciados e não devem ser usados em documentação nova:
- **expansion pack** → Use **squad**
- **permission mode** → Use **execution profile**
- **workflow state** → Use **flow-state**
</Callout>

### Termos Fundamentais do Framework

| Termo | Definição |
|---|---|
| **Agent** | Persona AI com expertise específica, definida em Markdown + YAML |
| **Task** | Instrução executável passo-a-passo que agentes seguem |
| **Template** | Estrutura YAML para geração de documentos com elicitação |
| **Workflow** | Sequência prescrita de steps e interações multi-agente |
| **Checklist** | Lista de validação que funciona como quality gate |
| **Agent Team** | Coleção pré-configurada de agentes para um propósito específico |
| **Elicitation** | Processo interativo de refinamento de conteúdo com o usuário |
| **Story** | Unidade atômica de trabalho com acceptance criteria e tasks |
| **Quality Gate** | Ponto de validação obrigatório entre fases do workflow |
| **Greenfield** | Projeto completamente novo, sem código existente |
| **Brownfield** | Projeto existente que recebe melhorias ou refactoring |
| **Document Sharding** | Fragmentação de documentos grandes em partes consumíveis |
| **Knowledge Base (KB)** | Base de conhecimento do framework em `aiox-kb.md` |
| **Technical Preferences** | Perfil de preferências técnicas que influencia recomendações |

### Semantic Lint Policy

O AIOX aplica **linting semântico** automatizado via `scripts/semantic-lint.js`:

- Termos depreciados em nível **error** bloqueiam commits e CI
- Termos em nível **warning** são reportados para migração gradual

Isso garante consistência terminológica em toda a documentação e código.

## Tech Stack

### Filosofia do Stack

A escolha tecnológica do AIOX segue dois princípios:

1. **Boring technology onde possível** -- dependências provadas e estáveis
2. **Exciting technology onde necessário** -- apenas quando performance ou DX justificam

### Core Runtime

```yaml
Node.js: 18.0.0+ (Active LTS)
Package Manager: npm 9.0.0+
Module System: CommonJS (ESM migration planejada)
```

**Por que Node.js 18+?**
- Native `fetch()` API -- sem necessidade de axios
- ES2022 support completo (top-level await, class fields)
- Cross-platform (Windows, macOS, Linux)
- Active LTS com patches de segurança

## Variáveis de Ambiente

```bash
# AIOX Configuration
AIOX_DEBUG=true                    # Habilita logs de debug
AIOX_CONFIG_PATH=/custom/path      # Caminho customizado para config
AIOX_YOLO_MODE=true               # Forca modo YOLO (sem confirmacoes)

# Node.js
NODE_ENV=production                # Modo de producao
NODE_OPTIONS=--max-old-space-size=4096  # Limite de memoria

# Integracoes
CLICKUP_API_KEY=pk_xxx            # ClickUp
GITHUB_TOKEN=ghp_xxx              # GitHub API
RAILWAY_TOKEN=xxx                 # Railway deploy
SUPABASE_ACCESS_TOKEN=xxx         # Supabase CLI
```

<Callout type="danger" title="Segurança">
NUNCA comite variáveis de ambiente com tokens ou API keys. Use `.env` (gitignored) e verifique seu `.gitignore` antes de cada push.
</Callout>
