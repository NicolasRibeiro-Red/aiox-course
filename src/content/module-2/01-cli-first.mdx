# CLI First: A Hierarquia de Prioridades

No mundo do desenvolvimento de software, a escolha da interface primária não é uma questão estética -- é uma **decisão arquitetural** que define a velocidade, portabilidade e reprodutibilidade de todo o sistema. O AIOX adota uma hierarquia clara e intencional: **CLI > Observability > UI**.

Nesta lição, você vai entender por que essa decisão foi tomada e como ela impacta cada aspecto do framework.

---

## A Hierarquia: CLI > Observability > UI

O AIOX opera com uma pirâmide de prioridades que inverte a lógica da maioria dos frameworks modernos:

| Nível | Prioridade | O que significa |
|---|---|---|
| **CLI** | Máxima | O terminal é a interface primária. Tudo que o framework faz, faz via linha de comando |
| **Observability** | Média | Métricas, logs e dashboards existem para monitorar, não para operar |
| **UI** | Mínima | Interfaces gráficas são opcionais e complementares, nunca obrigatórias |

<Callout type="info" title="Por que não UI First?">
Frameworks que dependem de UI criam fricção. Cada clique é tempo perdido. No AIOX, você ativa um agente com um comando e começa a trabalhar em segundos. O terminal é a interface mais rápida que existe para quem sabe o que quer fazer.
</Callout>

### Zero Overhead Visual

O AIOX não precisa de:
- Servidores web rodando localmente
- Dashboards obrigatórios consumindo recursos
- Interfaces gráficas pesadas para funcionar

O framework inteiro roda em **linguagem natural** -- Markdown e YAML. Sem binários, sem compilação, sem build steps obrigatórios para começar a trabalhar.

## O Diretório `aiox-core` como Cérebro

Toda a arquitetura do AIOX gira em torno do diretório `aiox-core`, que funciona como o cérebro central do sistema:

```
aiox-core/
├── agents/         # Definicoes de agentes (Markdown + YAML)
├── agent-teams/    # Times pre-configurados de agentes
├── workflows/      # Sequencias de trabalho multi-agente
├── templates/      # Templates para documentos (PRD, stories, etc.)
├── tasks/          # Instrucoes executaveis passo-a-passo
├── checklists/     # Checklists de validacao e qualidade
└── data/           # Knowledge base e preferencias tecnicas
```

Essa estrutura é **lida diretamente por LLMs** -- não precisa de parser, compilador ou runtime especial. Qualquer agente de IA que consiga ler texto pode consumir esses arquivos.

### O Web Builder

Para ambientes web, o AIOX inclui um builder (`tools/builders/web-builder.js`) que:

1. **Resolve dependências** -- lê a definição do agente ou time
2. **Encontra recursos** -- localiza tasks, templates e checklists referenciados
3. **Concatena tudo** -- gera um único arquivo `.txt` com separadores claros
4. **Salva em `dist/`** -- pronto para upload em qualquer interface web

```bash
# Gerar bundle para um time completo
node tools/builders/web-builder.js --team team-all

# Gerar bundle para um agente especifico
node tools/builders/web-builder.js --agent dev
```

## Agentes Dev vs. Agentes de Planejamento

Uma consequência direta do CLI First é a **separação clara** entre dois tipos de agentes:

| Característica | Agentes Dev (IDE) | Agentes de Planejamento (Web) |
|---|---|---|
| **Contexto** | Mínimo -- foco no código | Rico -- documentação completa |
| **Dependências** | Poucas tasks, lean | Múltiplas tasks e templates |
| **Objetivo** | Gerar código de qualidade | Criar PRDs, arquitetura, stories |
| **Exemplos** | `@dev`, `@qa`, `@devops` | `@pm`, `@architect`, `@analyst` |

<Callout type="tip" title="Regra de ouro do contexto">
Cada byte de contexto enviado a um agente Dev é um byte a menos disponível para entender seu código. Por isso agentes Dev são lean: carregam apenas o essencial para implementar a story atual.
</Callout>

### Contexto é Precioso

Em IDEs com agentes de IA, o tamanho da janela de contexto é limitado. O AIOX respeita isso:

- **Agentes Dev** carregam apenas a story atual, checklists de DoD e preferências técnicas
- **Agentes de planejamento** podem carregar PRDs completos, research, templates com elicitação

Essa economia de contexto é o que permite ao `@dev` ter espaço suficiente para "ver" o código do projeto inteiro em vez de desperdiçar tokens com documentação de planejamento.

## Integrável com Qualquer IDE

O CLI First significa que o AIOX não é preso a nenhum fornecedor:

| IDE / Plataforma | Como usar |
|---|---|
| **Claude Code** | Agentes via `.claude/commands/` |
| **Cursor** | Regras em `.cursor/rules/` |
| **Gemini CLI** | Agentes via `.codex/skills/` |
| **VS Code + Copilot** | Bundles `.txt` como contexto |
| **ChatGPT / Web** | Upload de bundles pré-compilados |

O framework se adapta ao ambiente -- não o contrário.

---

## Reprodutibilidade

Comandos no terminal são:

- **Determinísticos** -- o mesmo comando produz o mesmo resultado
- **Documentáveis** -- `*create-prd` é mais claro que "clique no botão azul, depois no menu..."
- **Automatizáveis** -- workflows inteiros podem ser scriptados
- **Versionáveis** -- toda a configuração vive em arquivos de texto no repositório

```bash
# Ativar agente e executar task -- reprodutivel, documentavel, automatizavel
@dev
*develop STORY-42
```
