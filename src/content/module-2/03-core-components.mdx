# Os 5 Building Blocks do AIOX

Todo o framework AIOX e construido sobre cinco componentes fundamentais: **Agents, Tasks, Templates, Workflows e Checklists**. Entender como cada um funciona -- e como se conectam entre si -- e essencial para usar o framework com eficacia. Nesta licao, vamos dissecar cada building block.

---

## Visao Geral da Arquitetura

```
aiox-core/
├── agents/         # 1. Quem faz o trabalho
├── tasks/          # 2. O que fazer (instrucoes)
├── templates/      # 3. Como documentar (estruturas)
├── workflows/      # 4. Em que ordem (sequencias)
├── checklists/     # 5. Esta bom? (validacao)
├── agent-teams/    # Agrupamentos de agentes
└── data/           # Knowledge base e preferencias
```

<Callout type="info" title="Tudo e linguagem natural">
Nenhum desses componentes e codigo executavel. Sao arquivos Markdown e YAML que LLMs leem e interpretam diretamente. Isso significa que voce pode inspecionar, editar e entender qualquer componente sem ferramentas especiais.
</Callout>

## 1. Agents (Agentes)

**Localizado em:** `aiox-core/agents/`

Agentes sao os **atores** do sistema. Cada agente e definido em um arquivo Markdown com frontmatter YAML que especifica:

| Atributo | Descricao | Exemplo |
|---|---|---|
| **Persona** | Nome e personalidade | Dex (The Builder) |
| **Archetype** | Papel arquetipico | Builder, Strategist, Guardian |
| **Capabilities** | O que sabe fazer | Implementar stories, criar PRDs |
| **Dependencies** | Tasks, templates e checklists que usa | `dev-develop-story.md`, `story-dod-checklist.md` |
| **Startup Instructions** | O que carregar ao ativar | Documentos do projeto, preferencias tecnicas |
| **Restrictions** | O que NAO pode fazer | @architect nao faz git push |

### Exemplo de Estrutura de Agente

```yaml
---
id: dev
name: Dex
title: Full Stack Developer
archetype: Builder
zodiac: Aquarius
icon: 💻
dependencies:
  tasks:
    - dev-develop-story.md
    - apply-qa-fixes.md
  checklists:
    - story-dod-checklist.md
  data:
    - technical-preferences.md
restrictions:
  - "NO git push -- delegate to @devops"
---

# Agent: Dex (@dev)

Dex is a pragmatic, solution-oriented full-stack developer...
```

### Agent Teams

**Localizado em:** `aiox-core/agent-teams/`

Times pre-configurados agrupam agentes para cenarios especificos:

| Time | Agentes | Uso |
|---|---|---|
| `team-all` | Todos os agentes | Bundle completo para web UI |
| `team-qa-focused` | @dev, @qa, @devops | Foco em qualidade e entrega |

## 2. Tasks (Tarefas)

**Localizado em:** `aiox-core/tasks/`

Tasks sao **instrucoes executaveis** -- procedimentos passo-a-passo que agentes seguem para completar uma acao especifica.

| Aspecto | Descricao |
|---|---|
| **Formato** | Markdown com passos numerados |
| **Escopo** | Uma acao atomica e reutilizavel |
| **Invocacao** | Via comandos `*` (ex: `*develop`, `*brainstorm`) |
| **Modos** | YOLO (autonomo), Interactive (padrao), Pre-Flight (planejamento completo) |

### Tasks Principais por Tipo

| Tipo | Exemplos | Agente Tipico |
|---|---|---|
| **Criacao** | `create-doc.md`, `create-next-story.md` | @pm, @sm |
| **Desenvolvimento** | `dev-develop-story.md`, `apply-qa-fixes.md` | @dev |
| **Analise** | `analyze-framework.md`, `spec-assess-complexity.md` | @architect, @analyst |
| **Validacao** | `execute-checklist.md`, `qa-review-story.md` | @qa, @po |
| **Operacoes** | `github-devops-pre-push-quality-gate.md` | @devops |

<Callout type="tip" title="Design de Tasks">
Prefira **multiplas tasks pequenas** a uma task grande com muitos branches. Tasks atomicas sao mais reutilizaveis e mais faceis de manter. A task `create-doc.md`, por exemplo, e usada por @pm, @architect e @analyst -- cada um com templates diferentes.
</Callout>

## 3. Templates

**Localizado em:** `aiox-core/templates/`

Templates sao **estruturas YAML** que definem o formato e conteudo de documentos gerados pelo framework.

### Tipos de Templates

| Template | Arquivo | Quem usa |
|---|---|---|
| **PRD** | `prd-tmpl.yaml` | @pm |
| **Architecture** | `fullstack-architecture-tmpl.yaml` | @architect |
| **User Story** | `story-tmpl.yaml` | @sm |
| **Project Brief** | `project-brief-tmpl.yaml` | @analyst |
| **Market Research** | `market-research-tmpl.yaml` | @analyst |
| **Competitor Analysis** | `competitor-analysis-tmpl.yaml` | @analyst |
| **Brownfield PRD** | `brownfield-prd-tmpl.yaml` | @pm |

### Anatomia de um Template

Templates AIOX sao **auto-contidos** -- embutem tanto a estrutura do documento quanto instrucoes para a LLM:

```yaml
template:
  id: prd-template-v2
  name: Product Requirements Document
  version: 2.0
  output:
    format: markdown
    filename: docs/prd.md

sections:
  - id: goals-context
    title: "Goals and Background"
    elicit: false
    instructions: "[[LLM: Extract goals from project brief...]]"

  - id: requirements
    title: "Requirements"
    elicit: true
    instructions: "[[LLM: Present elicitation options 1-9...]]"
```

A flag `elicit: true` indica que aquela secao requer interacao com o usuario -- o agente apresenta opcoes e aguarda resposta antes de prosseguir.

## 4. Workflows

**Localizado em:** `aiox-core/workflows/`

Workflows sao **sequencias prescritas** de steps e interacoes entre agentes para tipos especificos de projeto.

### Workflows Disponiveis

| Workflow | Tipo de Projeto | Agentes Envolvidos |
|---|---|---|
| `greenfield-fullstack` | Novo projeto full-stack | Todos |
| `greenfield-service` | Novo servico backend | @analyst -> @pm -> @architect -> @sm -> @dev |
| `greenfield-ui` | Nova interface frontend | @analyst -> @pm -> @ux -> @architect -> @sm -> @dev |
| `brownfield-fullstack` | Projeto existente full-stack | @analyst -> @architect -> @pm -> @sm -> @dev |
| `brownfield-service` | Servico existente | @architect -> @pm -> @sm -> @dev |
| `brownfield-ui` | Interface existente | @ux -> @architect -> @pm -> @sm -> @dev |

## 5. Checklists

**Localizado em:** `aiox-core/checklists/`

Checklists sao os **quality gates** do framework -- listas de validacao que garantem que artefatos estao completos e corretos antes de avancarem.

| Checklist | Quem usa | O que valida |
|---|---|---|
| `pm-checklist.md` | @pm, @po | 9 categorias de validacao de PRD |
| `architect-checklist.md` | @architect | Decisoes de arquitetura e trade-offs |
| `story-draft-checklist.md` | @sm | Completude e clareza de stories |
| `story-dod-checklist.md` | @dev | Definition of Done para implementacao |
| `po-master-checklist.md` | @po | Alinhamento geral entre documentos |
| `change-checklist.md` | @pm | Navegacao de mudancas mid-sprint |

### Status de Checklist

Cada categoria de um checklist resulta em um status:

| Status | Significado | Acao |
|---|---|---|
| **PASS** | 90%+ completo | Prosseguir para proxima fase |
| **PARTIAL** | 60-89% completo | Revisar itens pendentes |
| **FAIL** | Menos de 60% | Refazer secao antes de prosseguir |

## Como os 5 Blocos se Conectam

```
Agent ──usa──> Task ──processa──> Template ──gera──> Documento
  |                                                     |
  └──segue──> Workflow ──valida com──> Checklist ──────┘
```

Exemplo concreto:

1. `@pm` (Agent) executa `*create-prd` (Task)
2. A task processa `prd-tmpl.yaml` (Template) secao por secao
3. O resultado e `docs/prd.md` (Documento gerado)
4. `@po` valida com `pm-checklist.md` (Checklist)
5. Se PASS, o Workflow avanca para `@architect`

## Na Pratica: Artefatos Reais

Ate agora voce viu a teoria dos 5 building blocks. Agora vamos olhar artefatos **reais** extraidos diretamente do repositorio AIOX. Esses sao os arquivos que os agentes leem e executam.

### 1. Task: `create-doc.md` (definicao YAML)

Esta e a task que agentes como @pm e @architect usam para criar documentos a partir de templates. Note os tres modos de execucao (YOLO, Interactive, Pre-Flight) e como a task referencia ferramentas e utils:

```yaml
---
# Template selection determined dynamically during task execution
# User selects from available templates in .aiox-core/product/templates/
tools:
  - github-cli        # For file operations
utils:
  - template-engine
  - template-validator
---

# Create Document from Template (YAML Driven)

## Execution Modes

**Choose your execution mode:**

### 1. YOLO Mode - Fast, Autonomous (0-1 prompts)
- Autonomous decision making with logging
- Minimal user interaction
- **Best for:** Simple, deterministic tasks

### 2. Interactive Mode - Balanced, Educational (5-10 prompts) **[DEFAULT]**
- Explicit decision checkpoints
- Educational explanations
- **Best for:** Learning, complex decisions

### 3. Pre-Flight Planning - Comprehensive Upfront Planning
- Task analysis phase (identify all ambiguities)
- Zero ambiguity execution
- **Best for:** Ambiguous requirements, critical work

**Parameter:** `mode` (optional, default: `interactive`)
```

### 2. Template: `agent-handoff-tmpl.yaml` (completo)

Quando um agente passa o bastao para outro (ex: @dev → @qa), ele preenche este template de handoff. O artefato e compacto (max 500 tokens) e carrega apenas o essencial: story em andamento, decisoes tomadas, arquivos modificados e proxima acao:

```yaml
# Agent Handoff Artifact Template
# Purpose: Compact summary of previous agent's work for context-efficient agent switches.
# Constraint: Filled artifact MUST be < 500 tokens.
#
# Usage: On agent switch (@agent command), the outgoing agent populates this template.
#        The incoming agent receives: own full profile + this artifact (not previous agent's full persona).
#
# Reference: OpenAI Swarm handoff pattern

handoff:
  version: "1.0"
  timestamp: ""           # ISO 8601 (e.g., 2026-02-23T17:00:00Z)
  from_agent: ""          # Agent ID of outgoing agent (e.g., dev, qa, sm)
  to_agent: ""            # Agent ID of incoming agent (e.g., qa, devops)

  # What story/task is being worked on
  story_context:
    story_id: ""          # e.g., TOK-4A
    story_path: ""        # e.g., docs/stories/epics/epic-token-optimization/story-TOK-4A-...md
    story_status: ""      # e.g., In Progress, Ready for Review
    current_task: ""      # e.g., Task 3: Integration point & limits
    branch: ""            # e.g., feat/epic-token-optimization

  # Key decisions made by outgoing agent (max 5)
  decisions:
    - ""                  # e.g., "Used CLAUDE.md instructions for compaction (not code module)"

  # Files created or modified by outgoing agent (max 10)
  files_modified:
    - ""                  # e.g., ".aiox-core/development/templates/agent-handoff-tmpl.yaml"

  # Active blockers or issues (max 3)
  blockers:
    - ""                  # e.g., "NOG-18 not Done — Phase 2 SYNAPSE integration deferred"

  # What the incoming agent should do next (max 2 sentences)
  next_action: ""         # e.g., "Run QA gate on TOK-4A. Verify handoff preserves story context."

# --- Compaction Limits ---
# Max artifact size: 500 tokens (~375 words)
# Max retained summaries: 3 (oldest discarded on 4th switch)
# Max decisions: 5
# Max files_modified: 10
# Max blockers: 3
```

### 3. Checklist: `story-dod-checklist.md` (Definition of Done)

Este e o checklist que o @dev usa para validar se uma story esta realmente pronta antes de marcar como "Review". Note as instrucoes `[[LLM: ...]]` -- sao diretivas que a LLM interpreta durante a execucao:

```markdown
# Story Definition of Done (DoD) Checklist

## Instructions for Developer Agent

Before marking a story as 'Review', please go through each item in this checklist.
Report the status of each item (e.g., [x] Done, [ ] Not Done, [N/A] Not Applicable)
and provide brief comments if necessary.

[[LLM: INITIALIZATION INSTRUCTIONS - STORY DOD VALIDATION

This checklist is for DEVELOPER AGENTS to self-validate their work before marking
a story complete.

IMPORTANT: This is a self-assessment. Be honest about what's actually done vs what
should be done. It's better to identify issues now than have them found in review.

EXECUTION APPROACH:

1. Go through each section systematically
2. Mark items as [x] Done, [ ] Not Done, or [N/A] Not Applicable
3. Add brief comments explaining any [ ] or [N/A] items
4. Be specific about what was actually implemented
5. Flag any concerns or technical debt created

The goal is quality delivery, not just checking boxes.]]

## Checklist Items

1. **Requirements Met:**
   - [ ] All functional requirements specified in the story are implemented.
   - [ ] All acceptance criteria defined in the story are met.

2. **Coding Standards & Project Structure:**
   - [ ] All new/modified code strictly adheres to Operational Guidelines.
   - [ ] All new/modified code aligns with Project Structure.
   - [ ] Adherence to Tech Stack for technologies/versions used.
   - [ ] Adherence to Api Reference and Data Models.
   - [ ] Basic security best practices applied for new/modified code.
   - [ ] No new linter errors or warnings introduced.
   - [ ] Code is well-commented where necessary.

3. **Testing:**
   - [ ] All required unit tests are implemented.
   - [ ] All required integration tests (if applicable) are implemented.
   - [ ] All tests (unit, integration, E2E if applicable) pass successfully.
```
