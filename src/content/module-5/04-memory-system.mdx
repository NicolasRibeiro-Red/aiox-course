# Sistema de Memória — Como Agentes Lembram

O AIOX opera um sistema de memória em **duas camadas independentes** que permitem a agentes manter contexto entre sessões, aprender com erros e preservar decisões arquiteturais. Entender como essa memória funciona é essencial para usar o framework de forma eficaz.

---

## A Arquitetura de 2 Camadas

| Camada | Gerenciado Por | Escopo | Storage |
|---|---|---|---|
| **Camada 1: Claude Code Nativo** | Claude Code CLI | Auto Memory, CLAUDE.md, Transcripts | `~/.claude/` |
| **Camada 2: AIOX Framework** | Scripts JS em `.aiox-core/` | Gotchas, Snapshots, Timeline, Session State | `.aiox/` |

<Callout type="warning" title="Camadas independentes">
As duas camadas **não se comunicam entre si**. Cada uma persiste de forma independente. Isso é uma limitação arquitetural importante de entender.
</Callout>

## Camada 1: Claude Code Nativo

### Hierarquia de Carregamento

Toda nova sessão do Claude Code carrega automaticamente, nesta ordem:

| Ordem | Arquivo | Descrição |
|---|---|---|
| 1 | `~/.claude/CLAUDE.md` | Instruções globais do usuário |
| 2 | `Workspaces/.claude/CLAUDE.md` | Instruções do workspace |
| 3 | `projeto/.claude/CLAUDE.md` | Instruções do projeto |
| 4 | `projeto/.claude/rules/*.md` | Regras detalhadas |
| 5 | `~/.claude/projects/.../memory/MEMORY.md` | Auto Memory (primeiras 200 linhas) |

### Auto Memory — Leitura e Escrita

**Leitura (automática, toda sessão):**
- Claude Code lê o `MEMORY.md` ao iniciar
- Se tiver mais de 200 linhas, trunca — linhas 201+ são ignoradas
- O conteúdo é injetado no system prompt

**Escrita (durante sessão, decisão do Claude):**
- Quando Claude encontra um padrão, erro ou insight relevante, ele **decide** salvar
- Usa Write/Edit tool para atualizar o MEMORY.md
- Opcionalmente cria arquivos de tópico em `memory/compound-analysis/`

**Fim de sessão:**
- Transcrição salva como `.jsonl`
- **Nada mais acontece** — sem session-digest, sem memory-flush, sem sumarização automática

<Callout type="info" title="Limite de 200 linhas">
O MEMORY.md tem um limite prático de ~200 linhas. Além disso, o conteúdo é ignorado. Mantenha entradas concisas e use arquivos de tópico para detalhes.
</Callout>

### Agent Memory

Agentes definidos em `.claude/agents/` possuem memória individual:

```
.claude/agent-memory/
├── aiox-architect/MEMORY.md
├── aiox-dev/MEMORY.md
├── squad/MEMORY.md
└── ...
```

Cada agente carrega apenas seu próprio `MEMORY.md` ao ser ativado. Isso permite que agentes desenvolvam especialização e acumulem conhecimento específico do seu domínio.

## Camada 2: AIOX Framework

### 4 Módulos de Memória

| Módulo | Arquivo | Função |
|---|---|---|
| **Gotchas Memory** | `gotchas-memory.js` | Auto-captura: erro repetido 3x vira gotcha |
| **Context Snapshot** | `context-snapshot.js` | Salva/restaura contexto completo (max 50, 7 dias) |
| **File Evolution** | `file-evolution-tracker.js` | Histórico de mudanças por arquivo, detecta drift |
| **Timeline Manager** | `timeline-manager.js` | Facade unificada de todas as fontes, sync a cada 60s |

### Gotchas — Aprendendo com Erros

O sistema de gotchas captura automaticamente problemas recorrentes:

```
Erro acontece 1x → registrado no error-tracking.json
Erro acontece 2x → marcado como pattern
Erro acontece 3x → promovido para gotcha automaticamente
```

Gotchas ficam disponíveis para qualquer agente via `getContextForTask()`, prevenindo que o mesmo erro se repita.

Comandos:

```bash
*gotcha "Descricao do problema"   # Adicionar manualmente
*gotchas                          # Listar gotchas ativas
```

### Context Snapshots

Snapshots permitem salvar e restaurar o estado completo do contexto:

```javascript
// Salvar snapshot
const id = await snapshot.capture();

// Restaurar snapshot
await snapshot.restore(id);

// Listar snapshots
const list = await snapshot.list();

// Cleanup (max 50 snapshots, 7 dias)
await snapshot.cleanup();
```

Storage: `.aiox/snapshots/{id}.json`

### Timeline Manager

O Timeline Manager é uma **facade unificada** que integra todas as fontes de dados em uma única timeline:

```javascript
// Timeline unificada
const events = await timeline.getUnifiedTimeline();

// Adicionar evento manual
await timeline.addEvent({
  type: "decision",
  description: "Escolhido SQLite sobre PostgreSQL"
});

// Sync automatico a cada 60 segundos
timeline.autoSync();
```

## 3 Módulos de Sessão

| Módulo | Função |
|---|---|
| **Context Loader** | Carrega contexto relevante ao iniciar sessão |
| **Context Detector** | Detecta tipo de sessão (nova, continuação, retomada) |
| **Session State** | Persiste estado do epic/story atual |

O Context Loader é o Tier 3 do Unified Activation Pipeline — ele carrega contexto baseado no tipo de sessão detectado pelo Context Detector.

## Gaps e Limitações Conhecidas

É importante entender o que o sistema de memória **não faz**:

| Limitação | Impacto |
|---|---|
| Sem session-digest automático | Quando sessão fecha, nenhuma sumarização acontece |
| Sem memory-flush automático | MEMORY.md só atualiza durante a sessão |
| Camadas não se comunicam | `.claude/` e `.aiox/` são independentes |
| Limite de 200 linhas no MEMORY.md | Conteúdo além disso é ignorado |
| Hooks de sessão existem para Gemini, não Claude | Features de end-of-session não disponíveis |

<Callout type="tip" title="Boas práticas de memória">
Para compensar as limitações: mantenha o MEMORY.md conciso (use índices para tópicos), crie gotchas para erros recorrentes, e use snapshots antes de operações longas.
</Callout>

---

## Fluxo de Ativação de Agente (Memory Load)

Quando um agente é ativado, o sistema carrega sua memória em 3 tiers:

| Tier | O que carrega | Fonte |
|---|---|---|
| 1 | System prompt base | CLAUDE.md + rules |
| 2 | Persona e capabilities | Agent definition file |
| 3 | Contexto e memória | MEMORY.md + session context |

O Tier 3 usa o Context Loader para carregar gotchas relevantes, estado da sessão anterior e snapshots se disponíveis.

## Na Prática: Context Detection em Código

Abaixo está o trecho real do `greeting-builder.js` -- o script que constrói saudações inteligentes para cada agente. Ele mostra como o sistema carrega contexto (sessão, git, workflow, handoff) e combina tudo em menos de 200ms:

```javascript
/**
 * Greeting Builder - Context-Aware Agent Greeting System (Core Logic)
 *
 * Builds intelligent greetings based on:
 * - Session type (new/existing/workflow)
 * - Git configuration status
 * - Project status (natural language narrative)
 * - Command visibility metadata
 * - Previous agent handoff context
 * - Current story and branch references
 *
 * Performance: <200ms total (hard limit with timeout protection)
 * Fallback: Simple greeting on any error
 */

const ContextDetector = require('../../core/session/context-detector');
const GitConfigDetector = require('../../infrastructure/scripts/git-config-detector');
const WorkflowNavigator = require('./workflow-navigator');
const GreetingPreferenceManager = require('./greeting-preference-manager');
const { loadProjectStatus } = require('../../infrastructure/scripts/project-status-loader');
const { PermissionMode } = require('../../core/permissions');
const { resolveConfig } = require('../../core/config/config-resolver');
const { validateUserProfile } = require('../../infrastructure/scripts/validate-user-profile');
// SessionState integration for cross-terminal workflow continuity
const { SessionState } = require('../../core/orchestration/session-state');
// SurfaceChecker integration for proactive suggestions
const { SurfaceChecker } = require('../../core/orchestration/surface-checker');
const fs = require('fs');
const path = require('path');
const yaml = require('js-yaml');

const GREETING_TIMEOUT = 150; // 150ms hard limit per-section
const TOTAL_GREETING_TIMEOUT = 200; // 200ms total pipeline budget
const SECTION_TIMEOUT = 150; // 150ms per section builder

const DEFAULT_USER_PROFILE = 'advanced';
```

<Callout type="info" title="Imports revelam a arquitetura">
Os imports deste arquivo são um mapa da Camada 2 do sistema de memória. `ContextDetector` identifica o tipo de sessão, `GitConfigDetector` lê o estado do repositório, `WorkflowNavigator` sabe em que fase do workflow o agente está, e `SessionState` mantém continuidade entre terminais. Tudo isso alimenta o greeting em menos de 200ms.
</Callout>
