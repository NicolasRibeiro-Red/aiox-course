# Security Hardening — Protegendo seu Projeto AIOX

O AIOX opera em uma camada privilegiada entre modelos de IA e o sistema. Isso significa que segurança não é opcional — é fundamental. Este capítulo cobre as estratégias de hardening específicas para ambientes de desenvolvimento orquestrados por IA.

---

## Arquitetura de Segurança

O AIOX implementa **defesa em profundidade** com 4 camadas:

```
┌──────────────────────────────────────────────┐
│  CAMADA EXTERNA                               │
│  Firewall | TLS | Rate Limiting               │
├──────────────────────────────────────────────┤
│  CAMADA APLICAÇÃO                             │
│  Permission Modes | Input Validation          │
├──────────────────────────────────────────────┤
│  CAMADA EXECUÇÃO                              │
│  Sandboxing | Isolamento | Resource Limits    │
├──────────────────────────────────────────────┤
│  CAMADA DADOS                                 │
│  Encriptação | Storage Seguro | Audit Log     │
└──────────────────────────────────────────────┘
```

### Preocupações Específicas de IA

| Preocupação | Risco | Mitigação |
|---|---|---|
| Execução de código por agentes | CRÍTICO | Permission Modes, Sandboxing |
| Exposição de API keys | CRÍTICO | Isolamento de ambiente, encriptação |
| Command injection via IA | ALTO | Sanitização de input, hooks |
| Acesso não-autorizado a arquivos | ALTO | Restrições de diretório |
| Hijacking de sessão | MÉDIO | Rotação de tokens |
| Divulgação de informação | MÉDIO | Audit logging, controle de acesso |

## Gerenciamento de API Keys

API keys são os segredos mais críticos do AIOX. Chaves comprometidas podem levar a uso não-autorizado, vazamento de dados e impacto financeiro significativo.

### Hierarquia de Armazenamento

| Nível | Prática | Exemplo |
|---|---|---|
| **NUNCA** | Código fonte, repos Git, configs commitados, logs | Hardcoded no `.ts` |
| **Aceitável (Dev)** | Arquivos `.env` (gitignored), variáveis locais | `.env.local` |
| **Recomendado (Prod)** | Secret managers, CI/CD injection, Kubernetes secrets | AWS Secrets Manager |

<Callout type="danger" title="Regra #1 de segurança">
NUNCA commite API keys, tokens ou credenciais em repositórios Git. Adicione `.env`, `.env.local` e `*.pem` ao `.gitignore` IMEDIATAMENTE ao criar o projeto.
</Callout>

### Template Seguro de .env

```bash
# .env - NUNCA commitar este arquivo
# Adicionar a .gitignore: .env, .env.local, .env.*.local

# AI Provider Keys
ANTHROPIC_API_KEY=
OPENAI_API_KEY=

# MCP Server Keys
EXA_API_KEY=
GITHUB_TOKEN=
APIFY_TOKEN=

# Autenticacao
JWT_SECRET=        # Gerar com: openssl rand -hex 32
SESSION_SECRET=    # Gerar com: openssl rand -hex 32

# Encriptacao
DATABASE_ENCRYPTION_KEY=  # openssl rand -hex 32
```

### Política de Rotação de Chaves

| Tipo de Chave | Frequência | Em Caso de Comprometimento |
|---|---|---|
| AI Provider Keys | 90 dias | Imediato |
| JWT Secrets | 30 dias | Imediato |
| MCP Server Keys | 90 dias | Imediato |
| Service Tokens | 7 dias | Imediato |
| Dev Keys | Não reutilizar | Revogar imediatamente |

## PII Scanning — Proteção de Dados Pessoais

Antes de **qualquer commit** com conteúdo textual externo, execute um scan de PII (Personally Identifiable Information):

```bash
# Scan basico
grep -rn "CPF\|email@\|@gmail\|@hotmail" src/
grep -rn "sk-ant-\|sk-\|ghp_\|apify_api" src/

# Verificar historico Git
git log -p --all -S "API_KEY" -- .
git log -p --all -S "@gmail.com" -- .
```

| Tipo de PII | Pattern de Busca | Ação |
|---|---|---|
| Nomes completos | Patterns de nomes brasileiros | Remover ou anonimizar |
| Emails | `*@*.*` em conteúdo não-code | Remover |
| CPFs | `\d{3}\.\d{3}\.\d{3}-\d{2}` | NUNCA commitar |
| API Keys | `sk-`, `ghp_`, `apify_api_` | Revogar e rotacionar |
| Tokens | `Bearer `, `token=` | Mover para .env |

## Proteção de Branches

O AIOX configura proteção automática para branches críticas:

### Regras para `master`/`main`

| Proteção | Configuração |
|---|---|
| Status checks obrigatórios | ESLint, TypeScript, Tests, Story Validation |
| PR reviews obrigatórias | Mínimo 1 aprovação |
| Force push | **Bloqueado** |
| Deleção da branch | **Bloqueada** |
| Histórico linear | Sim (rebase only) |
| Dismiss stale reviews | Sim (em novos commits) |

```bash
# Configurar protecao automaticamente
node scripts/setup-branch-protection.js

# Verificar status
gh api repos/OWNER/REPO/branches/main/protection
```

## Validação de Input

Todo input de usuário e de IA deve ser validado antes de execução:

```javascript
// .aiox-core/core/security/key-validator.js
const requiredKeys = [
  { name: "ANTHROPIC_API_KEY", pattern: /^sk-ant-[a-zA-Z0-9_-]+$/ },
  { name: "JWT_SECRET", minLength: 32 },
];

function validateApiKeys() {
  const errors = [];
  for (const key of requiredKeys) {
    const value = process.env[key.name];
    if (!value) {
      errors.push(`Missing required key: ${key.name}`);
      continue;
    }
    if (key.pattern && !key.pattern.test(value)) {
      errors.push(`Invalid format for ${key.name}`);
    }
  }
  if (errors.length > 0) {
    throw new Error(`Validation Failed:\n${errors.join("\n")}`);
  }
}
```

## Isolamento de Ambientes

```javascript
// Validar isolamento de ambiente
function validateEnvironment() {
  // Impedir chaves de producao em desenvolvimento
  if (process.env.NODE_ENV === "development") {
    if (process.env.ANTHROPIC_API_KEY?.includes("prod")) {
      throw new Error("Production key in dev environment!");
    }
  }

  // Debug desligado em producao
  if (process.env.NODE_ENV === "production") {
    if (process.env.AIOX_DEBUG === "true") {
      console.warn("WARNING: Debug mode in production");
    }
  }
}
```

<Callout type="warning" title="Ambientes separados">
Nunca reutilize chaves entre ambientes. Mantenha conjuntos separados de API keys para desenvolvimento, staging e produção.
</Callout>

## Proteção de Arquivos de Secrets

```bash
# Criar diretorio seguro
mkdir -p ~/.aiox/secrets
chmod 700 ~/.aiox/secrets

# Encriptar arquivo de secrets
openssl enc -aes-256-cbc -salt -pbkdf2 \
  -in secrets.txt \
  -out ~/.aiox/secrets/encrypted.dat

# Permissoes corretas
chmod 600 ~/.aiox/secrets/*
```

## Audit Logging

Todas as operações sensíveis devem ser registradas:

```bash
# Configuracao de audit
AUDIT_LOG_ENABLED=true
AUDIT_LOG_PATH=/var/log/aiox/audit.log
AUDIT_LOG_RETENTION_DAYS=90
```

O AIOX registra automaticamente:
- Execução de comandos por agentes
- Acesso a arquivos sensíveis
- Mudanças em configuração de segurança
- Tentativas de autenticação
- Operações de Git (push, merge, delete)

## Checklist de Segurança

Use este checklist antes de colocar qualquer projeto AIOX em produção:

```markdown
### Pre-Deploy Security Checklist

# API Keys
- [ ] Todas as keys em secret manager (nao em .env)
- [ ] .env adicionado ao .gitignore
- [ ] Nenhuma key no historico do Git

# Branch Protection
- [ ] master/main protegida
- [ ] Force push bloqueado
- [ ] Reviews obrigatorias configuradas

# PII
- [ ] Scan de PII executado
- [ ] Nenhum dado pessoal no codigo
- [ ] Nenhum dado pessoal nos commits

# Ambiente
- [ ] Keys de dev separadas de prod
- [ ] Debug desligado em producao
- [ ] Audit logging habilitado

# Dependencias
- [ ] npm audit sem CRITICAL
- [ ] Dependencias atualizadas
- [ ] Lock file commitado
```
