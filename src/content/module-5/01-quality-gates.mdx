# Quality Gates — Arquitetura de 3 Camadas

O sistema de Quality Gates do AIOX fornece **garantia de qualidade automatizada** através de três camadas progressivas de validação. Cada camada captura tipos diferentes de problemas no estágio apropriado do desenvolvimento.

---

## A Arquitetura de 3 Camadas

| Camada | Tipo | Velocidade | Propósito |
|---|---|---|---|
| **Layer 1** | Automatizado (local) | ~30s | Capturar erros de sintaxe, linting e tipos |
| **Layer 2** | Assistido por IA | ~5 min | Capturar lógica, segurança e padrões |
| **Layer 3** | Humano | Variável | Review estratégico e sign-off final |

O princípio é simples: **quanto mais cedo você captura um problema, mais barato é corrigi-lo**. Layer 1 captura ~30% dos issues, Layer 2 adiciona ~50% (80% total), e Layer 3 pega os últimos ~20%.

---

## Layer 1: Pre-commit Checks

Checks rápidos e locais que rodam antes do código ser commitado. Capturam problemas óbvios imediatamente.

### Checks Incluídos

| Check | Ferramenta | Timeout | Descrição |
|---|---|---|---|
| **Lint** | ESLint | 60s | Estilo de código e boas práticas |
| **Test** | Jest | 5 min | Testes unitários com cobertura |
| **TypeCheck** | TypeScript | 2 min | Validação de tipos estática |

### Configuração

```yaml
# quality-gate-config.yaml
layer1:
  enabled: true
  failFast: true  # Para no primeiro erro
  checks:
    lint:
      enabled: true
      command: "npm run lint"
      failOn: "error"
      timeout: 60000
    test:
      enabled: true
      command: "npm test"
      timeout: 300000
      coverage:
        enabled: true
        minimum: 80
    typecheck:
      enabled: true
      command: "npm run typecheck"
      timeout: 120000
```

### Executando

```bash
# Todos os checks da Layer 1
aiox qa run --layer=1

# Check especifico
aiox qa run --layer=1 --check=lint
aiox qa run --layer=1 --check=test
```

## Layer 2: PR Automation

Review assistido por IA que roda em pull requests. Captura problemas mais profundos como erros de lógica, vulnerabilidades de segurança e problemas arquiteturais.

### Ferramentas Integradas

| Ferramenta | Propósito | Severidade Bloqueante |
|---|---|---|
| **CodeRabbit** | Review de código por IA | CRITICAL |
| **Quinn (@qa)** | Review automatizado de QA | CRITICAL |

### Níveis de Severidade

| Severidade | Ação | Descrição |
|---|---|---|
| **CRITICAL** | Bloqueia | Vulnerabilidade, risco de perda de dados, breaking change |
| **HIGH** | Warn + Documenta | Performance, validação faltando, anti-pattern |
| **MEDIUM** | Documenta | Code smell, sugestão de melhoria, risco menor |
| **LOW** | Ignora | Preferência de estilo, otimização menor |

<Callout type="danger" title="CRITICAL sempre bloqueia">
Issues de severidade CRITICAL sempre bloqueiam o merge. Eles tipicamente indicam vulnerabilidades de segurança, credenciais hardcoded, riscos de SQL injection ou XSS.
</Callout>

### Configuração

```yaml
layer2:
  enabled: true
  coderabbit:
    enabled: true
    command: "coderabbit --prompt-only -t uncommitted"
    timeout: 900000  # 15 minutos
    blockOn: [CRITICAL]
    warnOn: [HIGH]
    documentOn: [MEDIUM]
    ignoreOn: [LOW]
  quinn:
    enabled: true
    autoReview: true
    severity:
      block: [CRITICAL]
      warn: [HIGH, MEDIUM]
```

## Layer 3: Human Review

Review estratégico humano para sign-off final. Garante que requisitos de negócio foram atendidos e decisões arquiteturais são sólidas.

### Checklist de Review Estratégico

```markdown
### Arquitetura
- [ ] Mudanças alinhadas com arquitetura do sistema
- [ ] Sem dependências não-autorizadas
- [ ] Compatibilidade retroativa mantida

### Segurança
- [ ] Sem dados sensíveis expostos
- [ ] Validação de input presente
- [ ] Autenticação/autorização corretas

### Qualidade
- [ ] Código mantenível e legível
- [ ] Testes abrangentes
- [ ] Documentação atualizada

### Negócio
- [ ] Acceptance criteria atendidos
- [ ] Experiência do usuário considerada
- [ ] Performance aceitável
```

### Estratégias de Atribuição

| Estratégia | Descrição |
|---|---|
| **auto** | Atribui baseado em ownership de arquivos e expertise |
| **manual** | Atribuição manual do reviewer |
| **round-robin** | Rotação entre membros da equipe |

## CLI Completo

O AIOX fornece um CLI dedicado para quality gates:

```bash
# Rodar checks
aiox qa run                    # Todas as camadas
aiox qa run --layer=1          # Apenas Layer 1
aiox qa run --verbose          # Output detalhado
aiox qa run --fail-fast        # Para no primeiro erro

# Verificar status
aiox qa status                 # Status geral
aiox qa status --layer=2       # Status da Layer 2
aiox qa status --pr=123        # Status de um PR

# Gerar relatórios
aiox qa report                 # Relatório padrão
aiox qa report --format=markdown --output=qa-report.md

# Configurar
aiox qa configure              # Configuração interativa
aiox qa configure --layer1.coverage.minimum=90
```

## Integração com CI/CD

### GitHub Actions

```yaml
name: Quality Gate

on:
  pull_request:
    branches: [main, develop]

jobs:
  layer1:
    name: Layer 1 - Pre-commit
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "18"
      - run: npm ci
      - run: aiox qa run --layer=1

  layer2:
    name: Layer 2 - PR Automation
    needs: layer1
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: aiox qa run --layer=2
        env:
          CODERABBIT_API_KEY: ${{ secrets.CODERABBIT_API_KEY }}
```

### Pre-commit Hook (Husky)

```bash
# .husky/pre-commit
#!/bin/sh
aiox qa run --layer=1 --fail-fast
```

## Resolução de Problemas

| Camada | Problema | Solução |
|---|---|---|
| Layer 1 | Lint errors | `npm run lint -- --fix` para auto-fix |
| Layer 1 | Testes falhando | Verificar output, atualizar testes |
| Layer 1 | Timeout | Aumentar timeout na config |
| Layer 2 | CodeRabbit CRITICAL | Corrigir vulnerabilidade/breaking change |
| Layer 2 | CodeRabbit timeout | Verificar rede, tentar run manual |
| Layer 3 | Sem reviewer | Configurar `defaultReviewer` |
| Layer 3 | Sign-off expirado | Solicitar novo review |

## Na Prática: Configuração Real das Quality Gates

Este é o arquivo `quality-gate-config.yaml` completo que controla o comportamento das 3 camadas. Note como cada layer define seus checks, timeouts, severidades e estratégias de bloqueio:

```yaml
# Quality Gate Configuration
# Version: 1.0.0
# This configuration controls the behavior of the 3-layer Quality Gate system.
# Layers execute sequentially with fail-fast behavior.

version: "1.0"

# Layer 1: Pre-commit checks (fast, local)
layer1:
  enabled: true
  failFast: true  # Stop pipeline on first failure
  checks:
    lint:
      enabled: true
      command: "npm run lint"
      failOn: "error"  # error | warning
      timeout: 60000   # 1 minute
    test:
      enabled: true
      command: "npm test"
      timeout: 300000  # 5 minutes
      coverage:
        enabled: true
        minimum: 80
    typecheck:
      enabled: true
      command: "npm run typecheck"
      timeout: 120000  # 2 minutes

# Layer 2: PR Automation (external tools)
layer2:
  enabled: true
  coderabbit:
    enabled: true
    command: "coderabbit --prompt-only -t uncommitted"
    timeout: 900000  # 15 minutes
    blockOn:
      - CRITICAL
    warnOn:
      - HIGH
    documentOn:
      - MEDIUM
    ignoreOn:
      - LOW
  quinn:
    enabled: true
    autoReview: true
    agentPath: ".claude/commands/AIOX/agents/qa.md"
    severity:
      block: ["CRITICAL"]
      warn: ["HIGH", "MEDIUM"]

# Layer 3: Human Review (manual)
layer3:
  enabled: true
  requireSignoff: true
  assignmentStrategy: "auto"  # auto | manual | round-robin
  defaultReviewer: "@architect"
  checklist:
    enabled: true
    template: "strategic-review-checklist"
    minItems: 5
  signoff:
    required: true
    expiry: 86400000  # 24 hours in ms

# Report settings
reports:
  location: ".aiox/qa-reports"
  format: "json"  # json | yaml | markdown
  retention: 30   # days to keep reports
  includeMetrics: true

# Status persistence
status:
  location: ".aiox/qa-status.json"
  updateOnChange: true

# Verbose output settings
verbose:
  enabled: false
  showCommands: true
  showOutput: true
  showTimings: true
```

<Callout type="tip" title="Configuração, não código">
Todo o sistema de Quality Gates é controlado por este único YAML. Para ajustar o coverage mínimo de 80% para 90%, basta editar `layer1.checks.test.coverage.minimum`. Para desabilitar o CodeRabbit, mude `layer2.coderabbit.enabled` para `false`. Zero código necessário.
</Callout>
