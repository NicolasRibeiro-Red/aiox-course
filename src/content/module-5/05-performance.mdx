# Performance — Otimizando Projetos AIOX

Performance importa em dois níveis no AIOX: a velocidade dos **agentes durante a orquestração** e a velocidade do **código produzido**. Este capítulo foca em otimizações práticas que reduzem latência, evitam timeouts e tornam o ciclo de desenvolvimento mais fluido.

---

## Git fsmonitor — Aceleração de git status

### O Problema

O Unified Activation Pipeline do AIOX executa `git status` durante a ativação de agentes para detectar arquivos modificados e o estado do projeto. Em repositórios grandes, isso pode causar **timeouts no loader `projectStatus`** — uma taxa de timeout de ~60% foi observada em projetos com muitos arquivos.

### A Solução

O Git possui um recurso built-in chamado `fsmonitor` que usa um daemon de filesystem watcher para rastrear mudanças de arquivos. Em vez de escanear todo o working directory, ele mantém um cache atualizado em tempo real.

| Métrica | Sem fsmonitor | Com fsmonitor |
|---|---|---|
| Tempo de `git status` | 50-200ms | < 10ms |
| Taxa de timeout no Pipeline | ~60% | ~0% |
| Impacto na ativação de agentes | Perceptível | Imperceptível |

### Como Habilitar

```bash
# Habilitar para o repositorio atual
git config core.fsmonitor true

# Verificar se esta habilitado
git config core.fsmonitor
# Output: true
```

### Como Desabilitar

Se encontrar problemas (status stale, alto uso de CPU):

```bash
# Desabilitar
git config --unset core.fsmonitor

# Parar o daemon
git fsmonitor--daemon stop
```

### Diagnóstico

```bash
# Verificar status do fsmonitor
npx aiox-core doctor
```

O doctor reporta:
- **Habilitado**: `Git fsmonitor: enabled (git status acceleration active)`
- **Disponível**: `Git fsmonitor: not enabled (opt-in optimization available)`
- **Indisponível**: `Git fsmonitor: not available (Git 2.37+ required)`

### Requisitos e Limitações

| Requisito | Detalhe |
|---|---|
| Git | >= 2.37 (introduziu `fsmonitor--daemon`) |
| Filesystem | Local (não NFS, CIFS ou network-mounted) |
| OS | Windows (ReadDirectoryChangesW), macOS (FSEvents), Linux (inotify) |

<Callout type="warning" title="Filesystems de rede">
NÃO habilite fsmonitor em filesystems de rede (NFS, CIFS). O daemon pode não detectar mudanças remotas, causando `git status` com informações desatualizadas.
</Callout>

| Limitação | Descrição |
|---|---|
| Network filesystems | Daemon não detecta mudanças remotas |
| Repos muito grandes (> 100k arquivos) | Startup inicial pode levar segundos |
| WSL | Funciona, mas watchers cross-filesystem (Windows ↔ WSL) podem ter latência |

## Otimização de Pre-commit Hooks

Os pre-commit hooks do AIOX usam **cache incremental** para manter velocidade:

### ESLint com Cache

```json
{
  "cache": true,
  "cacheLocation": ".eslintcache"
}
```

- **Primeira execução**: ~10-15s (sem cache)
- **Execuções subsequentes**: < 5s (com cache)

O cache é invalidado quando: configuração muda, dependências são atualizadas, ou arquivos são deletados.

### TypeScript Incremental

```json
{
  "compilerOptions": {
    "incremental": true,
    "tsBuildInfoFile": ".tsbuildinfo"
  }
}
```

O compilador TypeScript reutiliza informações de compilações anteriores, verificando apenas arquivos que mudaram.

### .gitignore para Caches

Adicione ao `.gitignore` para não commitar caches:

```gitignore
.eslintcache
.tsbuildinfo
coverage/
```

## Otimização de CI/CD

### Jobs em Paralelo

O pipeline de CI do AIOX roda jobs em paralelo:

```yaml
jobs:
  lint:            # Roda em paralelo
  typecheck:       # Roda em paralelo
  test:            # Roda em paralelo
  story-validation: # Roda em paralelo
  validation-summary:
    needs: [lint, typecheck, test, story-validation]
```

Todos os jobs de validação rodam simultaneamente. O `validation-summary` só executa depois que todos completaram, agregando resultados.

### Cache de Dependências

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: "18"
    cache: "npm"  # Cache automatico de node_modules
```

O cache de npm evita reinstalar dependências a cada run.

### Timeouts por Job

| Job | Timeout | Justificativa |
|---|---|---|
| lint | 5 min | ESLint é rápido |
| typecheck | 5 min | TypeScript é rápido |
| test | 10 min | Testes podem ser mais lentos |
| story-validation | 5 min | Validação de YAML |

## Otimização de Validação de Stories

### Performance Atual

| Cenário | Tempo |
|---|---|
| Story única | < 100ms |
| Todas as stories | < 2s (típico) |

### Dicas de Otimização

1. **Stories focadas** — Uma feature por story reduz parsing
2. **Limitar tasks** — Quebrar stories grandes em menores
3. **YAML válido** — Erros de parsing são mais lentos que validação

## Boas Práticas Gerais de Performance

### Commits Pequenos e Frequentes

Menos arquivos por commit = validação mais rápida. Commits grandes com dezenas de arquivos sobrecarregam o ESLint e o TypeScript.

### Testar Durante Desenvolvimento

```bash
# Rode validacao manualmente antes de commitar
npm run lint
npm run typecheck
npm test
```

Encontrar e corrigir problemas durante o desenvolvimento é mais rápido do que esperar hooks ou CI falharem.

### Corrigir Issues Imediatamente

Não acumule lint warnings ou type errors. Cada issue não-corrigido adiciona tempo ao próximo commit.

### Manter Dependências Atualizadas

```bash
# Verificar dependencias desatualizadas
npm outdated

# Atualizar
npm update

# Verificar vulnerabilidades
npm audit
```

Dependências desatualizadas podem ser mais lentas e ter vulnerabilidades conhecidas.

## Monitoramento com aiox doctor

O comando `aiox doctor` faz um diagnóstico completo do ambiente:

```bash
npx aiox-core doctor
```

Ele verifica:
- Status do Git fsmonitor
- Versão do Git (>= 2.37 para fsmonitor)
- Versão do Node.js
- Estado dos caches (ESLint, TypeScript)
- Configuração de hooks (Husky)
- Saúde das dependências

<Callout type="tip" title="Rode periodicamente">
Execute `aiox doctor` pelo menos 1x por semana ou quando notar lentidão. Ele identifica problemas de performance antes que se tornem críticos.
</Callout>

| Área | Otimização | Impacto |
|---|---|---|
| git status | Habilitar fsmonitor | 50-200ms → < 10ms |
