# Contribuindo para o AIOX Open Source

O AIOX é um projeto open source que cresce com contribuições da comunidade. Nesta lição, você vai aprender o processo completo para contribuir — de um fix simples até a criação de squads inteiras.

---

## Tipos de Contribuição

| Tipo | Descrição | Dificuldade |
|------|-----------|-------------|
| **Documentação** | Corrigir typos, melhorar guias | Fácil |
| **Bug Fixes** | Corrigir issues reportados | Fácil-Médio |
| **Tasks** | Adicionar novos workflows de tarefas | Médio |
| **Agents** | Criar novas personas de agentes IA | Médio |
| **Squads** | Pacote de agentes + tasks + workflows | Avançado |
| **Core Features** | Melhorias no framework | Avançado |

<Callout type="tip" title="Comece Pequeno">
Se é sua primeira contribuição, comece com documentação ou bug fixes. Isso ajuda a entender o codebase e o processo de review antes de trabalhar em features maiores.
</Callout>

## Quick Start

### 1. Fork e Clone

```bash
# Fork via GitHub UI, depois clone seu fork
git clone https://github.com/SEU_USERNAME/aiox-core.git
cd aiox-core

# Adicionar remote upstream
git remote add upstream https://github.com/SynkraAI/aiox-core.git
```

### 2. Setup do Ambiente

**Pré-requisitos:** Node.js >= 20.0.0, npm, Git

```bash
# Instalar dependencias
npm install

# Verificar setup
npm test
npm run lint
npm run typecheck
```

### 3. Criar Branch

```bash
git checkout -b feature/your-feature-name
```

**Convenções de branch:**

| Prefixo | Uso |
|---------|-----|
| `feature/` | Novas features, agentes, tasks |
| `fix/` | Bug fixes |
| `docs/` | Atualizações de documentação |
| `refactor/` | Refatoração de código |
| `test/` | Adições/melhorias de testes |

### 4. Fazer Mudanças

Siga o guia relevante abaixo para cada tipo de contribuição.

### 5. Validação Local

```bash
npm run lint      # Estilo de codigo
npm run typecheck # Verificacao de tipos
npm test          # Rodar testes
npm run build     # Verificar build
```

### 6. Push e PR

```bash
git push origin feature/your-feature-name
```

Depois crie um Pull Request no GitHub targeting a branch `main`.

## Contribuindo Agentes

Agentes são o tipo de contribuição mais comum. Cada agente é um arquivo Markdown com YAML frontmatter.

### Localização

```
.aiox-core/development/agents/your-agent.md
```

### Estrutura Obrigatória

```yaml
agent:
  name: AgentName
  id: agent-id           # kebab-case, unico
  title: Titulo Descritivo
  icon: emoji
  whenToUse: "Quando ativar"

persona_profile:
  archetype: Builder | Analyst | Guardian | Operator | Strategist
  communication:
    tone: pragmatic | friendly | formal | analytical

commands:
  - help: Mostrar comandos
  - custom: Descricao

dependencies:
  tasks:
    - related-task.md
```

### Checklist de Validação

- [ ] Agent ID é único e usa kebab-case
- [ ] `persona_profile` completo com archetype e communication
- [ ] Todos os commands têm descrição
- [ ] Dependencies listam todas as tasks necessárias
- [ ] Sem credenciais ou dados sensíveis hardcoded
- [ ] Segue padrões existentes no codebase

## Contribuindo Tasks

Tasks são workflows executáveis que agentes podem rodar.

### Localização

```
.aiox-core/development/tasks/your-task.md
```

### Estrutura Obrigatória

```markdown
# Task Name

**Description:** O que a task faz
**Agent(s):** @dev, @qa
**Elicit:** true | false

---

## Prerequisites
- Prerequisito 1

## Steps
### Step 1: Primeiro Passo
Descricao detalhada.

## Deliverables
- [ ] Entregavel 1

## Error Handling
Se X acontecer, faca Y.
```

## Contribuindo Squads

Squads são a contribuição mais complexa — pacotes completos de agentes + tasks + workflows.

### Estrutura

```
your-squad/
├── squad.yaml        # Manifesto
├── agents/
│   └── your-agent.md
├── tasks/
│   └── your-task.md
└── workflows/
    └── your-workflow.yaml
```

### Manifesto Mínimo

```yaml
name: your-squad
version: 1.0.0
description: O que este squad faz
author: Seu Nome
dependencies:
  - base-squad (opcional)
agents:
  - your-agent
tasks:
  - your-task
```

## Processo de Code Review

### Checks Automáticos

Ao submeter um PR, os seguintes checks rodam automaticamente:

| Check | Descrição | Obrigatório |
|-------|-----------|-------------|
| **ESLint** | Estilo e qualidade | Sim |
| **TypeScript** | Type checking | Sim |
| **Jest Tests** | Suite de testes (Node 18 e 20) | Sim |
| **Validation Summary** | Gate agregado | Sim |
| **Build** | Verificação de build | Não (advisory) |
| **Coverage** | Relatório de cobertura | Não (advisory) |

### CodeRabbit AI Review

O [CodeRabbit](https://coderabbit.ai) revisa automaticamente seu PR e fornece feedback sobre:

- Qualidade de código e melhores práticas
- Preocupações de segurança
- Padrões específicos do AIOX
- Issues de performance

| Severidade | Ação Necessária |
|-----------|-----------------|
| **CRITICAL** | Obrigatório corrigir antes do merge |
| **HIGH** | Fortemente recomendado corrigir |
| **MEDIUM** | Considerar corrigir ou documentar como tech debt |
| **LOW** | Melhoria opcional |

### Review de Maintainer

Após os checks automáticos, um maintainer:
1. Verifica se as mudanças atendem os padrões
2. Checa implicações de segurança
3. Garante que documentação está atualizada
4. Aprova ou solicita mudanças

## Sistema de Validação — Defense in Depth

O AIOX implementa 3 camadas de validação:

### Camada 1: Pre-commit (Local)

Performance: < 5 segundos

- ESLint com cache
- TypeScript compilação incremental
- IDE sync (auto-stage arquivos de comando)

### Camada 2: Pre-push (Local)

Performance: < 2 segundos

- Validação de checkboxes de stories
- Checks de consistência de status

### Camada 3: CI/CD (Cloud)

Performance: 2-5 minutos

- Lint e type checking completos
- Suite de testes completa
- Relatório de cobertura
- Validação de stories

## Convenções de Commit

O AIOX usa [Conventional Commits](https://www.conventionalcommits.org/):

```bash
# Exemplos
git commit -m "feat(agent): add security-auditor agent"
git commit -m "fix: resolve memory leak in config loader"
git commit -m "docs: update contribution guide"
```

**Tipos:** `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

## Branch Protection

Todas as mudanças na `main` passam por Pull Request. Push direto é bloqueado.

### Requisitos para Merge

- [ ] Todos os CI checks passam
- [ ] Pelo menos 1 aprovação de maintainer
- [ ] Todas as conversas resolvidas
- [ ] Sem conflitos de merge
- [ ] Branch atualizada com main

### CODEOWNERS

Paths críticos requerem aprovação de maintainers específicos:

| Path | Razão |
|------|-------|
| `.aiox-core/core/orchestration/` | Camada de orquestração |
| `.aiox-core/core/execution/` | Engine de execução |
| `packages/` | Installer, CLI, libs |
| `.github/` | CI/CD workflows |

## Trabalhando com Pro

O AIOX usa modelo Open Core com um submódulo privado `pro/`:

```bash
# Para contribuidores open-source:
# Voce NAO precisa do submodulo pro/
git clone https://github.com/SynkraAI/aiox-core.git
cd aiox-core
npm install && npm test  # Tudo funciona sem pro/
```

O diretório `pro/` simplesmente não existirá no seu clone — isso é esperado e todos os testes passam sem ele.

## FAQ

**P: Quanto tempo leva o review?**
R: Buscamos primeiro review em 24-48 horas. Mudanças complexas podem demorar mais.

**P: Posso contribuir sem testes?**
R: Testes são fortemente encorajados. Para mudanças somente de docs, testes podem não ser necessários.

**P: Meu PR tem conflitos. O que faço?**
```bash
git fetch upstream
git rebase upstream/main
git push --force-with-lease
```

**P: Posso contribuir em português?**
R: Sim! PRs em português são aceitos.
