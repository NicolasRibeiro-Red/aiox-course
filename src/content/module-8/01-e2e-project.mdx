# Construindo um Task Tracker com AIOX — Walkthrough Completo

Nesta licao, voce vai acompanhar a construcao de uma aplicacao **do zero ao deploy** usando o AIOX. Em vez de explicar conceitos abstratos, vamos simular as interacoes reais entre voce e os agentes — exatamente como acontece no terminal.

O projeto: um **Task Tracker** — uma aplicacao web para gerenciar tarefas com autenticacao, boards e colaboracao em tempo real. Simples o bastante para caber em um walkthrough, complexo o bastante para exercitar todos os agentes.

<Callout type="info" title="Como Ler Este Walkthrough">
Cada bloco de terminal abaixo simula uma interacao real com um agente AIOX. As mensagens "voce" sao o que voce digitaria. As respostas dos agentes sao baseadas no comportamento documentado de cada um. Artefatos mostram trechos reais dos documentos gerados.
</Callout>

## O Ciclo Completo

Vamos recapitular o que aconteceu — 7 fases, 7 agentes, um fluxo continuo:

```
Fase 0: @devops    → Bootstrap do ambiente        → environment-report.json
Fase 1: @analyst   → Discovery e project brief    → project-brief.md
Fase 2: @pm        → Product Requirements         → prd.md
Fase 3: @architect → Arquitetura tecnica          → fullstack-architecture.md
Fase 4: @sm        → Stories do backlog           → 1.1.story.md
Fase 5: @dev       → Implementacao + testes       → codigo + testes
Fase 6: @qa        → Quality gate                 → gate-file.yml (PASS)
Fase 7: @devops    → Push + deploy                → live na Vercel
```

### Pontos-chave do Walkthrough

1. **Cada agente tem um papel claro** — nenhum faz o trabalho do outro. Atlas pesquisa, Morgan planeja, Aria projeta, River fragmenta, Dex constroi, Quinn valida, Gage opera.

2. **Documentos fluem entre agentes** — o output de um e o input do proximo. Project Brief → PRD → Arquitetura → Stories → Codigo → Gate.

3. **Quality gates sao reais** — o Quinn nao e rubber stamp. Ele identifica issues, classifica severidade e decide se bloqueia ou documenta.

4. **Decisoes tecnicas sao documentadas** — quando o Dex escolhe Server Actions em vez de API routes, ele registra o porquê. Quando a Aria escolhe Zustand, ela justifica.

5. **O ciclo repete** — apos a Story 1.1, o `@sm` cria a 1.2, o `@dev` implementa, o `@qa` revisa. O loop continua ate o MVP estar completo.

<Callout type="tip" title="Na Pratica">
Em um projeto real, cada fase pode acontecer em sessoes separadas. O AIOX persiste tudo em arquivos — voce pode fechar o terminal, abrir outro dia, e continuar de onde parou. Sempre commite ao final de cada fase.
</Callout>
