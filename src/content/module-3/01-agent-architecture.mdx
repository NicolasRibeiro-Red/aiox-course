# Arquitetura de Agentes

Os agentes são a alma do AIOX. Cada um é mais do que um prompt -- é uma **persona completa** com expertise, personalidade, dependências e regras de colaboração. Nesta lição, você vai entender como agentes são construídos, como se ativam e como se comunicam entre si.

---

## Anatomia de um Agente

Todo agente AIOX é definido em um arquivo Markdown com frontmatter YAML. Essa definição inclui:

| Componente | Descrição | Exemplo |
|---|---|---|
| **Persona** | Nome, título e personalidade | Dex -- The Builder |
| **Archetype** | Papel arquetípico (inspirado em arquétipos jungianos) | Builder, Strategist, Guardian |
| **Zodiac** | Signo associado (parte da persona) | Aquarius, Capricorn, Virgo |
| **Icon** | Ícone visual para identificação rápida | 💻, 📋, 🧪 |
| **Dependencies** | Tasks, templates e checklists que o agente utiliza | `dev-develop-story.md`, `story-dod-checklist.md` |
| **Commands** | Lista de comandos `*` disponíveis | `*develop`, `*help`, `*exit` |
| **Restrictions** | O que o agente NÃO pode fazer | "NO git push" |
| **Signature** | Frase de encerramento característica | "-- Dex, sempre construindo" |

### Vocabulário e Tom

Cada agente tem vocabulário e tom de comunicação distintos:

| Agente | Tom | Vocabulário Chave |
|---|---|---|
| **Dex** (@dev) | Pragmático, conciso | Construir, implementar, refatorar, otimizar |
| **Morgan** (@pm) | Estratégico, analítico | Planejar, priorizar, alinhar, escalar |
| **Quinn** (@qa) | Sistemático, educacional | Validar, verificar, garantir, proteger |
| **River** (@sm) | Empático, colaborativo | Adaptar, pivotar, simplificar, fluir |
| **Aria** (@architect) | Holístico, pragmático | Projetar, integrar, escalar, otimizar |
| **Atlas** (@analyst) | Analítico, inquisitivo | Pesquisar, investigar, descobrir, sintetizar |

## O Sistema de Agentes Completo

O AIOX possui 12 agentes organizados em 5 camadas:

| Camada | Agentes | Função |
|---|---|---|
| **Orquestração** | Orion (@aiox-master) | Meta-agente que coordena todos |
| **Discovery e Planejamento** | Atlas (@analyst), Morgan (@pm) | Pesquisa, requisitos, estratégia |
| **Design e Arquitetura** | Aria (@architect), Uma (@ux-design-expert), Dara (@data-engineer) | Arquitetura técnica, UX, dados |
| **Gestão** | Pax (@po), River (@sm) | Backlog, stories, sprints |
| **Execução** | Dex (@dev), Quinn (@qa), Gage (@devops), Craft (@squad-creator) | Código, testes, deploy, squads |

## Pipeline de Ativação (Unified Activation Pipeline)

Quando você ativa um agente com `@agent-name`, o seguinte pipeline é executado:

| Step | O que acontece |
|---|---|
| 1. **Load Agent** | Arquivo de definição do agente é carregado |
| 2. **Load Dependencies** | Tasks, checklists e dados referenciados são carregados |
| 3. **Context Detection** | Pipeline detecta projeto atual, stories em progresso, estado do git |
| 4. **Contextual Greeting** | Saudação inteligente baseada no contexto detectado |
| 5. **Quick Commands** | Lista os 3-6 comandos mais relevantes para o contexto atual |
| 6. **HALT** | Aguarda o primeiro comando do usuário |

### Sistema de Saudação Inteligente

O greeting system não é um "Olá!" genérico. Ele analisa o contexto e adapta:

```
# Exemplo: Ativando @dev em projeto com story em andamento

🔨 Dex online.

Story ativa: STORY-42 (Implement User Auth)
  Tasks: 2/4 completas
  Ultimo arquivo: src/auth/login.ts

Quick commands:
  *develop STORY-42  -- continuar implementacao
  *help              -- ver todos os comandos
  *status            -- ver estado atual

-- Dex, sempre construindo
```

<Callout type="info" title="O agente NÃO age sozinho">
Após o greeting, o agente sempre faz HALT e aguarda o comando do usuário. Nenhuma ação é tomada automaticamente na ativação.
</Callout>

## Modos de Execução

Todos os agentes suportam três modos de execução:

| Modo | Interação | Ideal para |
|---|---|---|
| **YOLO** | 0-1 prompts, decisões autônomas | Tasks simples e determinísticas |
| **Interactive** (padrão) | 5-10 prompts, checkpoints de decisão | Aprendizado, decisões complexas |
| **Pre-Flight** | Planejamento completo antes de executar | Requisitos ambíguos, trabalho crítico |

Para alternar o modo YOLO:

```
*yolo    # Toggle on/off
```

## Diagrama de Relacionamentos

Os agentes não trabalham isolados. Existe um fluxo claro de artefatos entre eles:

| De | Para | Artefato | Gatilho |
|---|---|---|---|
| @analyst | @pm | project-brief.md, market-research.md | Pesquisa completa |
| @pm | @architect | prd.md | PRD aprovado (pm-checklist PASS) |
| @pm | @ux-design-expert | prd.md (seções UI) | PRD com requisitos de UI |
| @ux-design-expert | @architect | front-end-spec.md | Spec de frontend completa |
| @architect | @data-engineer | Requisitos de schema | Arquitetura data-heavy |
| @architect | @dev | architecture.md | Arquitetura aprovada |
| @po | @sm | Backlog priorizado | Epics validados |
| @sm | @dev | STORY-*.md | Story aprovada pelo usuário |
| @dev | @qa | Código implementado | Story marcada "ready for review" |
| @qa | @dev | qa-report.md (com fix requests) | Issues encontradas no review |
| @dev | @devops | Código pronto para deploy | QA aprovado |

## Restrições e Boundaries

Cada agente tem limites claros sobre o que pode e não pode fazer:

| Agente | FAZ | NÃO FAZ |
|---|---|---|
| @architect | Seleciona database, projeta APIs | Git push (delega para @devops) |
| @dev | Implementa stories, escreve testes | Cria stories (recebe do @sm) |
| @qa | Revisa código, gera testes | Implementa fixes (cria fix-request para @dev) |
| @data-engineer | Schema, migrations, RLS | Seleção de tecnologia de DB (cabe ao @architect) |
| @devops | Git push, PR, CI/CD | Implementação de features |
| @analyst | Pesquisa, brainstorming | Criação de PRD (cabe ao @pm) |

<Callout type="tip" title="Boundaries evitam conflitos">
As restrições existem para que não haja duplicação de responsabilidade. Se dois agentes pudessem fazer a mesma coisa, nenhum faria direito.
</Callout>

## Na Prática: Definição Real de um Agente

Abaixo está a definição **real** do agente Atlas (@analyst), extraída diretamente do repositório AIOX. Observe como o YAML define persona, arquétipo, signo, níveis de greeting, vocabulário, comandos e princípios fundamentais -- tudo em linguagem natural que a LLM interpreta:

```yaml
agent:
  name: Atlas
  id: analyst
  title: Business Analyst
  icon: 🔍
  whenToUse: |
    Use for market research, competitive analysis, user research,
    brainstorming session facilitation, structured ideation workshops,
    feasibility studies, industry trends analysis, project discovery
    (brownfield documentation), and research report creation.

    NOT for: PRD creation or product strategy → Use @pm.
    Technical architecture decisions → Use @architect.
    Story creation or sprint planning → Use @sm.

persona_profile:
  archetype: Decoder
  zodiac: '♏ Scorpio'

  communication:
    tone: analytical
    emoji_frequency: minimal

    vocabulary:
      - explorar
      - analisar
      - investigar
      - descobrir
      - decifrar
      - examinar
      - mapear

    greeting_levels:
      minimal: '🔍 analyst Agent ready'
      named: "🔍 Atlas (Decoder) ready. Let's uncover insights!"
      archetypal: '🔍 Atlas the Decoder ready to investigate!'

    signature_closing: '— Atlas, investigando a verdade 🔎'

persona:
  role: Insightful Analyst & Strategic Ideation Partner
  style: Analytical, inquisitive, creative, facilitative, objective, data-informed
  identity: Strategic analyst specializing in brainstorming, market research,
            competitive analysis, and project briefing
  core_principles:
    - Curiosity-Driven Inquiry
    - Objective & Evidence-Based Analysis
    - Strategic Contextualization
    - Facilitate Clarity & Shared Understanding
    - Creative Exploration & Divergent Thinking
    - Structured & Methodical Approach
    - Action-Oriented Outputs
    - Collaborative Partnership
    - Maintaining a Broad Perspective
    - Integrity of Information
    - Numbered Options Protocol

commands:
  - name: help
    visibility: [full, quick, key]
    description: 'Show all available commands with descriptions'
  - name: create-project-brief
    visibility: [full, quick]
    description: 'Create project brief document'
```

<Callout type="tip" title="Persona como especificação">
Note como `whenToUse` define não apenas QUANDO usar o agente, mas também QUANDO NÃO usar -- redirecionando para o agente correto. Essa clareza de boundaries é o que evita conflitos entre agentes.
</Callout>
