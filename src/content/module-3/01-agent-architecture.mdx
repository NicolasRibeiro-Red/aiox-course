# Arquitetura de Agentes

Os agentes sao a alma do AIOX. Cada um e mais do que um prompt -- e uma **persona completa** com expertise, personalidade, dependencias e regras de colaboracao. Nesta licao, voce vai entender como agentes sao construidos, como se ativam e como se comunicam entre si.

---

## Anatomia de um Agente

Todo agente AIOX e definido em um arquivo Markdown com frontmatter YAML. Essa definicao inclui:

| Componente | Descricao | Exemplo |
|---|---|---|
| **Persona** | Nome, titulo e personalidade | Dex -- The Builder |
| **Archetype** | Papel arquetipico (inspirado em arquetipos jungianos) | Builder, Strategist, Guardian |
| **Zodiac** | Signo associado (parte da persona) | Aquarius, Capricorn, Virgo |
| **Icon** | Icone visual para identificacao rapida | 💻, 📋, 🧪 |
| **Dependencies** | Tasks, templates e checklists que o agente utiliza | `dev-develop-story.md`, `story-dod-checklist.md` |
| **Commands** | Lista de comandos `*` disponiveis | `*develop`, `*help`, `*exit` |
| **Restrictions** | O que o agente NAO pode fazer | "NO git push" |
| **Signature** | Frase de encerramento caracteristica | "-- Dex, sempre construindo" |

## Pipeline de Ativacao (Unified Activation Pipeline)

Quando voce ativa um agente com `@agent-name`, o seguinte pipeline e executado:

| Step | O que acontece |
|---|---|
| 1. **Load Agent** | Arquivo de definicao do agente e carregado |
| 2. **Load Dependencies** | Tasks, checklists e dados referenciados sao carregados |
| 3. **Context Detection** | Pipeline detecta projeto atual, stories em progresso, estado do git |
| 4. **Contextual Greeting** | Saudacao inteligente baseada no contexto detectado |
| 5. **Quick Commands** | Lista os 3-6 comandos mais relevantes para o contexto atual |
| 6. **HALT** | Aguarda o primeiro comando do usuario |

### Sistema de Saudacao Inteligente

O greeting system nao e um "Ola!" generico. Ele analisa o contexto e adapta:

```
# Exemplo: Ativando @dev em projeto com story em andamento

🔨 Dex online.

Story ativa: STORY-42 (Implement User Auth)
  Tasks: 2/4 completas
  Ultimo arquivo: src/auth/login.ts

Quick commands:
  *develop STORY-42  -- continuar implementacao
  *help              -- ver todos os comandos
  *status            -- ver estado atual

-- Dex, sempre construindo
```

<Callout type="info" title="O agente NAO age sozinho">
Apos o greeting, o agente sempre faz HALT e aguarda o comando do usuario. Nenhuma acao e tomada automaticamente na ativacao.
</Callout>

## Modos de Execucao

Todos os agentes suportam tres modos de execucao:

| Modo | Interacao | Ideal para |
|---|---|---|
| **YOLO** | 0-1 prompts, decisoes autonomas | Tasks simples e deterministicas |
| **Interactive** (padrao) | 5-10 prompts, checkpoints de decisao | Aprendizado, decisoes complexas |
| **Pre-Flight** | Planejamento completo antes de executar | Requisitos ambiguos, trabalho critico |

Para alternar o modo YOLO:

```
*yolo    # Toggle on/off
```

## Diagrama de Relacionamentos

Os agentes nao trabalham isolados. Existe um fluxo claro de artefatos entre eles:

| De | Para | Artefato | Gatilho |
|---|---|---|---|
| @analyst | @pm | project-brief.md, market-research.md | Pesquisa completa |
| @pm | @architect | prd.md | PRD aprovado (pm-checklist PASS) |
| @pm | @ux-design-expert | prd.md (secoes UI) | PRD com requisitos de UI |
| @ux-design-expert | @architect | front-end-spec.md | Spec de frontend completa |
| @architect | @data-engineer | Requisitos de schema | Arquitetura data-heavy |
| @architect | @dev | architecture.md | Arquitetura aprovada |
| @po | @sm | Backlog priorizado | Epics validados |
| @sm | @dev | STORY-*.md | Story aprovada pelo usuario |
| @dev | @qa | Codigo implementado | Story marcada "ready for review" |
| @qa | @dev | qa-report.md (com fix requests) | Issues encontradas no review |
| @dev | @devops | Codigo pronto para deploy | QA aprovado |

## Restricoes e Boundaries

Cada agente tem limites claros sobre o que pode e nao pode fazer:

| Agente | FAZ | NAO FAZ |
|---|---|---|
| @architect | Seleciona database, projeta APIs | Git push (delega para @devops) |
| @dev | Implementa stories, escreve testes | Cria stories (recebe do @sm) |
| @qa | Revisa codigo, gera testes | Implementa fixes (cria fix-request para @dev) |
| @data-engineer | Schema, migrations, RLS | Selecao de tecnologia de DB (cabe ao @architect) |
| @devops | Git push, PR, CI/CD | Implementacao de features |
| @analyst | Pesquisa, brainstorming | Criacao de PRD (cabe ao @pm) |

<Callout type="tip" title="Boundaries evitam conflitos">
As restricoes existem para que nao haja duplicacao de responsabilidade. Se dois agentes pudessem fazer a mesma coisa, nenhum faria direito.
</Callout>
