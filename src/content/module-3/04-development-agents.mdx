# Agentes de Desenvolvimento: Dex, Dara e Uma

Estes tres agentes transformam planejamento em realidade. **Dex** constroi codigo, **Dara** arquiteta dados e **Uma** projeta experiencias de usuario. Cada um e especialista profundo no seu dominio, com ferramentas e workflows especificos.

### O Workflow de Desenvolvimento

Quando Dex recebe `*develop STORY-42`, ele segue este fluxo:

| Step | Acao | Descricao |
|---|---|---|
| 1 | Read Story | Carrega STORY-42.md completa |
| 2 | Load Context | Carrega arquitetura, technical preferences, gotchas |
| 3 | Plan Approach | Analisa tasks e define ordem de implementacao |
| 4 | Implement | Executa cada task sequencialmente |
| 5 | Write Tests | Testes para cada acceptance criteria |
| 6 | Update Story | Marca tasks como `[x]`, atualiza file list |
| 7 | Self-Review | Verifica DoD checklist |
| 8 | Ready for Review | Marca story como pronta para QA |

### Story DoD (Definition of Done)

Antes de marcar uma story como completa, Dex valida:

- Todos os acceptance criteria implementados
- Testes escritos e passando
- Lint sem erros (`npm run lint`)
- Types sem erros (`npm run typecheck`)
- File list atualizado na story
- Tasks marcadas como `[x]`

### O Loop QA

Quando @qa encontra issues, Dex recebe um `qa-report.md` e usa `*apply-qa-fixes`:

```
@qa encontra issue → qa-report.md → @dev *apply-qa-fixes → @qa verifica fix
```

O loop continua ate que @qa emita PASS no quality gate.

<Callout type="info" title="Recovery System">
Se Dex falha em implementar uma subtask, ele tem um sistema de recovery: `*track-attempt` registra a tentativa, e apos 3 falhas, `*rollback` reverte para o ultimo estado bom.
</Callout>

### Principios Criticos de Dara

| Principio | Significado |
|---|---|
| **Corretude antes de velocidade** | Acertar primeiro, otimizar depois |
| **Tudo versionado e reversivel** | Snapshots + scripts de rollback |
| **Seguranca por padrao** | RLS, constraints, triggers |
| **Idempotencia em todo lugar** | Seguro executar operacoes multiplas vezes |
| **Access pattern first** | Design baseado em como dados serao consultados |
| **Zero-downtime como objetivo** | Migracoes planejadas cuidadosamente |

### Workflow de Migracao Segura

```
1. *snapshot "pre-migration"     -- Salva estado atual
2. *dry-run migration.sql        -- Testa sem commit
3. *apply-migration migration.sql -- Aplica com lock advisory
4. *smoke-test v2                -- Testes abrangentes
5. (se falhar) *rollback "pre-migration"
```

### Databases Suportados

| Database | Comando | Especialidade |
|---|---|---|
| **Supabase** | `*setup-database supabase` | PostgreSQL + Auth + RLS |
| **PostgreSQL** | `*setup-database postgresql` | Relacional puro |
| **MongoDB** | `*setup-database mongodb` | Documentos |
| **MySQL** | `*setup-database mysql` | Relacional classico |
| **SQLite** | `*setup-database sqlite` | Embarcado |

### As 5 Fases de Uma

| Fase | Foco | Comandos | Output |
|---|---|---|---|
| **1. UX Research** | Pesquisa de usuario, personas, jornadas | `*research`, `*wireframe`, `*create-front-end-spec` | Personas, wireframes, front-end-spec.md |
| **2. Design System Audit** | Escanear redundancias em codebase existente | `*audit {path}`, `*consolidate`, `*shock-report` | Relatorio de redundancias HTML |
| **3. Design Tokens** | Extrair e configurar tokens | `*tokenize`, `*setup`, `*migrate` | Tokens configurados |
| **4. Component Building** | Construir atoms, molecules, organisms | `*build {component}`, `*compose {molecule}`, `*extend` | Componentes atomicos |
| **5. Quality & Docs** | Acessibilidade, documentacao, ROI | `*a11y-check`, `*document`, `*calculate-roi` | Pattern library, WCAG report |

### Atomic Design no AIOX

Uma segue a metodologia Atomic Design de Brad Frost:

| Nivel | Descricao | Exemplo |
|---|---|---|
| **Atoms** | Elementos basicos indivisiveis | Button, Input, Label |
| **Molecules** | Grupos de atoms com funcao | SearchBar (Input + Button) |
| **Organisms** | Secoes complexas da interface | Header (Logo + Nav + SearchBar) |

<Callout type="tip" title="Audit antes de Build">
Em projetos brownfield, SEMPRE execute `*audit {path}` antes de criar novos componentes. Uma identifica redundancias existentes que podem ser consolidadas em vez de criar duplicatas.
</Callout>
