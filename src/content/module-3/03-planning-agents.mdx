# Agentes de Planejamento: Atlas, Morgan e Aria

Antes de uma única linha de código ser escrita, três agentes trabalham para garantir que o projeto tenha fundação sólida: **Atlas** investiga, **Morgan** planeja e **Aria** projeta a arquitetura. Juntos, transformam uma ideia vaga em documentação técnica acionável.

### Artefatos que Atlas Produz

| Artefato | Comando | Destino |
|---|---|---|
| Project Brief | `*create-project-brief` | Input para @pm criar PRD |
| Market Research | `*perform-market-research` | Dados de mercado (TAM/SAM/SOM) |
| Competitor Analysis | `*create-competitor-analysis` | Análise SWOT, Blue Ocean |
| Brainstorming Results | `*brainstorm {topic}` | Ideias categorizadas e priorizadas |
| Research Prompt | `*research-prompt {topic}` | Prompt estruturado para pesquisa profunda |

### Fluxo de Brainstorming

O brainstorming de Atlas segue 4 fases:

| Fase | O que acontece | Duração |
|---|---|---|
| **1. Setup** | 4 perguntas de contexto (topic, constraints, goal, format) | 5 min |
| **2. Approach** | Seleção de técnicas (usuário escolhe, Atlas recomenda, aleatório ou progressivo) | 2 min |
| **3. Execution** | Ciclo interativo: aplicar técnica -> engajar -> capturar ideias -> repetir | 20-30 min |
| **4. Convergence** | Categorizar ideias, sintetizar insights, gerar documento final | 10-15 min |

<Callout type="tip" title="Quantidade antes de qualidade">
Atlas segue a regra: gerar muitas ideias primeiro, avaliar depois. O alvo é 100 ideias em 60 minutos. Julgamento é adiado para a fase de convergência.
</Callout>

### O Fluxo do PRD

Morgan cria PRDs em 8 passos:

| Step | Seção | Elicit? | Descrição |
|---|---|---|---|
| 1 | Goals and Background | Não | Objetivos e contexto extraídos do brief |
| 2 | Requirements | **Sim** | Requisitos funcionais e não-funcionais |
| 3 | UI Design Goals | **Sim** | Visão UX/UI (condicional -- só se tem UI) |
| 4 | Technical Assumptions | **Sim** | Decisões técnicas iniciais |
| 5 | Epic List | **Sim** | Lista de epics para aprovação da estrutura |
| 6 | Epic Details | **Sim** | Stories e acceptance criteria detalhados |
| 7 | Checklist Results | Não | Resultado do pm-checklist (9 categorias) |
| 8 | Next Steps | Não | Prompts para @architect e @ux-expert |

### Correção de Curso

Quando algo muda no meio do desenvolvimento, Morgan usa `*correct-course` para navegar a mudança:

1. **Entender o gatilho** -- O que mudou e por quê?
2. **Avaliar impacto em epics** -- Quais epics são afetados?
3. **Analisar conflitos** -- PRD, Architecture, Frontend Spec precisam de update?
4. **Avaliar opções** -- Ajustar, rollback ou re-scope?
5. **Gerar Sprint Change Proposal** -- Documento formal com plano de ação

### Princípios de Arquitetura

Aria segue 10 princípios que guiam todas as decisões:

| # | Princípio | Na prática |
|---|---|---|
| 1 | Holistic System Thinking | Cada componente é parte de um sistema maior |
| 2 | User Experience Drives Architecture | Começa com jornadas do usuário |
| 3 | Pragmatic Technology Selection | "Boring" onde possível, "exciting" onde necessário |
| 4 | Progressive Complexity | Simples no início, escalável depois |
| 5 | Cross-Stack Performance | Otimização em todas as camadas |
| 6 | Developer Experience | Produtividade do dev é prioridade |
| 7 | Security at Every Layer | Defesa em profundidade |
| 8 | Data-Centric Design | Dados guiam a arquitetura |
| 9 | Cost-Conscious Engineering | Ideais vs. realidade financeira |
| 10 | Living Architecture | Projetada para mudança |

### Avaliação de Complexidade

Aria usa 5 dimensões para avaliar a complexidade de uma story:

| Dimensão | Score 1 | Score 5 |
|---|---|---|
| **Scope** | 1-2 arquivos | 20+ arquivos |
| **Integration** | Nenhuma integração externa | Orquestração complexa |
| **Infrastructure** | Nenhuma mudança de infra | Novo servidor necessário |
| **Knowledge** | Padrões existentes | Domínio desconhecido |
| **Risk** | Impacto baixo | Impacto crítico |

**Score total (5-25):**
- **SIMPLE** (até 8): menos de 1 dia
- **STANDARD** (9-15): 1-3 dias
- **COMPLEX** (16+): 3+ dias
