# Agentes de Operações: Gage e Craft

Os agentes de operações garantem que o código chega ao mundo real e que o framework se expande de forma organizada. **Gage** cuida de infraestrutura, CI/CD e deploy. **Craft** cria squads modulares para novos domínios.

### Regra Crítica: Governança MCP

<Callout type="danger" title="MCP = Exclusivo do DevOps">
Toda operação de infraestrutura MCP (Model Context Protocol) é gerenciada EXCLUSIVAMENTE pelo Gage. Outros agentes são CONSUMIDORES de MCP, não administradores. Precisa adicionar um MCP? Chame @devops.
</Callout>

| Operação | Comando | Descrição |
|---|---|---|
| Buscar MCPs | `*search-mcp` | Pesquisa catálogo de servidores MCP disponíveis |
| Adicionar MCP | `*add-mcp` | Instala e configura novo servidor MCP |
| Listar MCPs | `*list-mcps` | Lista servidores MCP habilitados |
| Remover MCP | `*remove-mcp` | Remove servidor MCP |
| Setup Docker | `*setup-mcp-docker` | Configura MCPs via Docker MCP Toolkit |

### Os 4 Domínios de Gage

#### 1. Git e GitHub Operations

| Comando | Descrição |
|---|---|
| `*push` | Pre-push quality gate + git push |
| `*setup-github` | Configura repositório GitHub |
| `*cleanup` | Limpeza de repositório (branches, tags) |
| `*version-check` | Verifica e gerencia versões |

O pre-push quality gate (`*push`) executa automaticamente:
- `npm run lint` -- Linting
- `npm run typecheck` -- Verificação de tipos
- `npm test` -- Testes automatizados
- Security scan -- Busca por secrets expostos

#### 2. CI/CD e Releases

| Comando | Descrição |
|---|---|
| `*ci-cd` | Configura pipeline de integração e entrega contínua |
| `*release` | Gerencia versionamento e publicação de releases |
| `*environment-bootstrap` | Bootstrap completo de novo ambiente |

#### 3. Segurança

| Comando | Descrição |
|---|---|
| `*security-audit` | Auditoria completa de segurança |
| `*security-scan` | Varredura rápida por vulnerabilidades |

#### 4. MCP Management

Todos os comandos `*search-mcp`, `*add-mcp`, `*list-mcps`, `*remove-mcp`, `*setup-mcp-docker`.

### Autoridade Exclusiva sobre Git Remoto

Este é um ponto crítico do AIOX: **somente @devops pode interagir com repositórios remotos**.

| Operação | Quem pode |
|---|---|
| `git push` | @devops APENAS |
| `gh pr create` | @devops APENAS |
| `gh pr merge` | @devops APENAS |
| `git add/commit` | @dev, @sm |
| `git status/log/diff` | Todos |

### Worktree Management

Gage também gerencia worktrees para desenvolvimento paralelo:

| Comando | Descrição |
|---|---|
| `*create-worktree {story}` | Cria worktree Git isolada para story |
| `*list-worktrees` | Lista worktrees ativas |
| `*merge-worktree {story}` | Merge worktree completa de volta ao main |
| `*cleanup-worktrees` | Remove worktrees stale/merged |

<Callout type="tip" title="Worktrees para trabalho paralelo">
Worktrees permitem trabalhar em múltiplas stories simultaneamente em branches isoladas, sem conflitos de git. Cada story tem seu próprio diretório de trabalho.
</Callout>

### O que são Squads?

Squads são **extensões modulares** do AIOX para domínios especializados. Cada squad é um pacote autocontido com:

```
squads/my-squad/
├── squad.yaml          # Manifest obrigatorio
├── README.md           # Documentacao
├── agents/             # Agentes especializados do dominio
├── tasks/              # Tasks especificas
├── workflows/          # Workflows de orquestracao
└── config/             # Configuracoes
```

### Quando Criar um Squad?

| Cenário | Criar Squad? | Razão |
|---|---|---|
| Domínio não-técnico (wellness, educação) | Sim | Vai além de dev de software |
| Domínio técnico especializado (games, mobile) | Sim | Agentes específicos necessários |
| Base de conhecimento pesada | Sim | Bloataria o core |
| Feature simples de software | Não | Core agents resolvem |
| Workflow repetitivo no seu time | Sim | Reutilizável entre projetos |

### Princípios Fundamentais

| Princípio | Descrição |
|---|---|
| **Task-First Architecture** | Tasks são o ponto de entrada, agentes orquestram |
| **Validação obrigatória** | Sempre validar contra JSON Schema antes de distribuir |
| **3 níveis de distribuição** | Local, Público (aiox-squads), Marketplace (Synkra API) |
| **Integração com aiox-core** | Squads trabalham em sinergia com o framework |

### 3 Níveis de Distribuição

| Nível | Onde vive | Escopo | Comando |
|---|---|---|---|
| **Local** | `./squads/` no projeto | Apenas este projeto | `*create-squad` |
| **Público** | Repositório `aiox-squads` | Qualquer projeto AIOX | `*publish-squad` (futuro) |
| **Marketplace** | Synkra API | Ecossistema global | `*sync-squad-synkra` (futuro) |

### Validação Obrigatória

Craft valida squads contra JSON Schema antes de qualquer distribuição:

```bash
*validate-squad my-squad

# squad.yaml: Schema valido
# agents/: 3 agentes com persona completa
# tasks/: 7 tasks com comandos mapeados
# workflows/: 1 workflow de orquestracao
# config/: Faltam defaults para 2 variaveis (warning)

Resultado: PASS (com warnings)
```
