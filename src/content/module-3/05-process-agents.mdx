# Agentes de Processo: River, Quinn e Pax

Estes três agentes garantem que o trabalho flua com qualidade: **River** facilita o processo ágil e cria stories, **Quinn** protege a qualidade do código e **Pax** gerencia o backlog e valida artefatos. Sem eles, o pipeline de desenvolvimento perderia ritmo e consistência.

### O Papel Central de River

River é o **elo entre planejamento e execução**. Ele transforma documentos de alto nível (PRD, epics, arquitetura) em stories atômicas que @dev pode implementar:

```
PRD + Epic + Architecture  →  @sm  →  STORY-42.md  →  @dev implementa
```

### Anatomia de uma Story Criada por River

Quando River executa `*draft`, ele gera uma story com:

| Seção | Conteúdo |
|---|---|
| **Title** | Título claro e descritivo |
| **Description** | Contexto, objetivo e valor de negócio |
| **Tasks** | Lista ordenada de subtasks com checkboxes `[ ]` |
| **Acceptance Criteria** | Formato Given-When-Then testável |
| **Architecture Reference** | Links para seções relevantes da arquitetura |
| **Risk Profile** | Avaliação de risco da story |
| **Agent Assignment** | Quais agentes serão necessários |
| **File List** | Arquivos que serão criados/modificados |

### Story Draft Checklist

Antes de entregar uma story para @dev, River valida com o `story-draft-checklist.md`:

- Título é descritivo e autoexplicativo?
- Tasks são atômicas e sequenciais?
- Acceptance criteria são testáveis (Given-When-Then)?
- Referências de arquitetura estão corretas?
- Risco foi avaliado?
- Agentes necessários foram identificados?

<Callout type="warning" title="Story = Contrato">
A story é um contrato entre @sm e @dev. Se a story não é clara o suficiente, @dev vai precisar pesquisar -- e isso desperdiça tempo e contexto precioso.
</Callout>

## Quinn -- A Guardiã (@qa)

<AgentCard
  icon="🧪"

  archetype="Guardian"
  zodiac="Virgo"

  commands={["*gate {story}", "*review {story}", "*test-design {story}", "*risk-profile {story}", "*nfr-assess {story}", "*trace {story}", "*generate-tests", "*run-tests", "*backlog-add", "*create-fix-request {story}"]}
  whenToUse="Após implementação de stories. Quando precisa de code review, design de testes, avaliação de risco, validação de NFRs, ou geração automática de testes."
  collaboratesWith={["@dev (envia fix requests, recebe fixes)", "@sm (coordena risk profiling)", "@architect (escala issues arquiteturais)", "@devops (valida antes de deploy)"]}
  restrictions={["NÃO implementa código (cria fix-request para @dev)", "NÃO faz deploy", "NÃO cria stories"]}
/>

### Os 8 Princípios de Quinn

| # | Princípio | Na prática |
|---|---|---|
| 1 | Profundidade conforme necessário | Vai fundo em alto risco, rápido em baixo risco |
| 2 | Rastreabilidade | Cada story mapeia para testes Given-When-Then |
| 3 | Testes baseados em risco | Prioriza por probabilidade x impacto |
| 4 | Atributos de qualidade | Valida NFRs (segurança, performance, confiabilidade) |
| 5 | Avaliação de testabilidade | Controlabilidade, observabilidade, debuggabilidade |
| 6 | Governança de gates | Decisões claras PASS/CONCERNS/FAIL/WAIVED |
| 7 | Excelência consultiva | Educa, nunca bloqueia arbitrariamente |
| 8 | Integração CodeRabbit | Revisão automatizada para detectar issues cedo |

### Quality Gate: As 4 Decisões

Quando Quinn executa `*gate {story}`, o resultado é uma das 4 decisões:

| Decisão | Significado | Ação |
|---|---|---|
| **PASS** | Tudo aprovado, qualidade confirmada | Story pode seguir para deploy |
| **CONCERNS** | Aprovado com ressalvas documentadas | Story pode seguir, mas com debt registrado |
| **FAIL** | Issues críticos encontrados | Story volta para @dev com fix-request |
| **WAIVED** | Gate dispensado com justificativa | Aceito conscientemente (raro) |

### O Loop QA

```
@dev implementa → @qa *review {story} → FAIL?
    ↓                                      ↓
  PASS → deploy                    @qa *create-fix-request
                                          ↓
                                   @dev *apply-qa-fixes
                                          ↓
                                   @qa *review (re-review)
                                          ↓
                                   Loop ate PASS
```

### Design de Testes

Quinn não só revisa -- ele projeta testes estruturados:

```markdown
## Test Case: User Login

**Given** a registered user with valid credentials
**When** the user submits login form with email and password
**Then** a JWT token is returned with 200 status

**Given** a user with invalid password
**When** the user submits login form
**Then** a 401 error is returned with message "Invalid credentials"
```

### O Master Checklist

O artefato mais importante de Pax é o **PO Master Checklist** -- a validação final que garante alinhamento entre todos os documentos antes do desenvolvimento começar:

| O que valida | Documentos envolvidos |
|---|---|
| PRD está completo e claro? | `prd.md` |
| Arquitetura reflete o PRD? | `fullstack-architecture.md` + `prd.md` |
| UX spec está alinhada? | `front-end-spec.md` + `prd.md` |
| Epics cobrem todo o escopo? | Epics vs. PRD requirements |
| Stories são acionáveis? | Stories vs. acceptance criteria |

### Document Sharding

Quando documentos ficam grandes demais para o contexto de agentes Dev, Pax usa `*shard-doc` para fragmentá-los:

```
docs/prd.md (5000 palavras)
    ↓  *shard-doc
docs/prd/
├── epic-1-auth.md
├── epic-2-dashboard.md
├── epic-3-payments.md
└── epic-4-notifications.md
```

Cada fragmento é autocontido e pode ser carregado individualmente por @sm e @dev.

<Callout type="tip" title="Sharding é obrigatório">
Na transição de Web UI para IDE, Pax DEVE fragmentar documentos grandes. Agentes de desenvolvimento não conseguem processar PRDs de 5000+ palavras sem estourar o contexto.
</Callout>
