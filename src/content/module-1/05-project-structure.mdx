# Estrutura do Projeto

Entender a estrutura de diretorios do AIOX e fundamental para trabalhar efetivamente com o framework. Cada pasta tem um proposito claro, e saber onde cada coisa vive evita confusao e acelera seu workflow.

---

## Visao Geral

O AIOX usa uma **arquitetura modular** com separacao clara de responsabilidades:

1. **Framework Core** (`.aiox-core/`) -- Componentes portaveis organizados por dominio
2. **Project Workspace** (raiz) -- Implementacao especifica do projeto
3. **Documentation** (`docs/`) -- Documentacao publica e stories

A filosofia e: **organizacao por dominio**, nao por tipo de arquivo. Agentes ficam com agentes, templates ficam com templates, infraestrutura fica com infraestrutura.

## Modulos em Detalhe

### `.aiox-core/development/` -- Assets de Desenvolvimento

Este e o coracao do framework. Aqui vivem os agentes, tasks e workflows.

**Agentes** (`.aiox-core/development/agents/`):

Cada agente e um arquivo Markdown com frontmatter YAML que define persona, comandos e dependencias. Sao 11 agentes core:

| Arquivo | Agente | Papel |
|---|---|---|
| `aiox-master.md` | @aiox-master (Orion) | Orquestrador Mestre |
| `dev.md` | @dev (Dex) | Desenvolvedor Senior |
| `qa.md` | @qa (Quinn) | Arquiteto de Testes |
| `architect.md` | @architect (Aria) | Arquiteta de Sistemas |
| `pm.md` | @pm (Morgan) | Product Manager |
| `po.md` | @po (Pax) | Product Owner |
| `sm.md` | @sm (River) | Scrum Master |
| `analyst.md` | @analyst (Atlas) | Analista de Negocios |
| `data-engineer.md` | @data-engineer (Dara) | Engenheira de Dados |
| `devops.md` | @devops (Gage) | Guardiao de Repositorios |
| `ux-design-expert.md` | @ux-design-expert (Uma) | Designer UX/UI |

**Tasks** (`.aiox-core/development/tasks/`):

Mais de 60 tasks disponiveis. Cada task e um procedimento step-by-step que um agente executa. Exemplos:

- `create-next-story.md` -- Cria a proxima story do backlog
- `develop-story.md` -- Implementa uma story
- `spec-write-spec.md` -- Escreve especificacao tecnica

**Workflows** (`.aiox-core/development/workflows/`):

Workflows orquestram multiplas tasks em sequencia:

- `greenfield-fullstack.yaml` -- Projeto novo do zero
- `brownfield-service.yaml` -- Feature em projeto existente
- `spec-pipeline.yaml` -- Pipeline de especificacao (ADE)
- `qa-loop.yaml` -- Loop de review e correcao (ADE)

| `elicitation/` | Engine de prompting interativo |

| `registry/` | Registro de servicos |

<Callout type="info" title="Voce raramente edita core/">
O diretorio core/ e gerenciado pelo framework. Voce interage com ele indiretamente atraves de agentes e configuracao. A menos que esteja contribuindo para o AIOX, nao modifique esses arquivos.
</Callout>

### `.aiox-core/data/` -- Dados Compartilhados

Arquivos de dados que agentes leem durante a ativacao:

| Arquivo | Dono | Funcao |
|---|---|---|
| `aiox-kb.md` | @aiox-master | Knowledge base do AIOX (lazy-loaded) |
| `agent-config-requirements.yaml` | @architect | Regras de carregamento por agente |
| `technical-preferences.md` | @architect | Preferencias tecnicas do time |
| `workflow-patterns.yaml` | @sm | Padroes de deteccao de workflow |

Cada arquivo tem um **dono** (agente responsavel) e um **trigger de atualizacao** definido. Isso garante que dados compartilhados nao fiquem desatualizados.

## Convencoes de Nomenclatura

O AIOX segue regras estritas de nomenclatura:

```yaml
Diretorios: kebab-case
  .aiox-core/, development/, quality-gates/

Arquivos de codigo: kebab-case + extensao
  agent-executor.js, task-runner.js

Arquivos de docs: kebab-case + .md
  coding-standards.md, story-6.1.2.5.md

Templates: {nome}-tmpl.{yaml|md}
  story-tmpl.yaml, prd-tmpl.md

Checklists: {nome}-checklist.md
  architect-checklist.md

Stories: story-{epic}.{story}.{substory}.md
  story-6.1.2.5.md

Epics: epic-{numero}-{nome}.md
  epic-6.1-agent-identity-system.md
```

<Callout type="warning" title="Siga as convencoes">
Arquivos fora do padrao podem nao ser detectados pelo framework. Se voce criar um agente chamado `MyAgent.MD` em vez de `my-agent.md`, ele pode nao ser carregado corretamente.
</Callout>

## Onde Colocar Novos Arquivos

Uma referencia rapida para quando voce precisar criar algo:

| Voce quer criar... | Coloque em... |
|---|---|
| Novo agente | `.aiox-core/development/agents/{nome}.md` |
| Nova task | `.aiox-core/development/tasks/{nome}.md` |
| Novo workflow | `.aiox-core/development/workflows/{nome}.yaml` |
| Novo template | `.aiox-core/product/templates/{nome}-tmpl.yaml` |
| Nova checklist | `.aiox-core/product/checklists/{nome}-checklist.md` |
| Novo CLI command | `.aiox-core/cli/commands/{categoria}/{nome}.js` |
| Nova story | `docs/stories/{contexto}/{nome}.md` |
| Novo teste | `tests/{tipo}/{nome}.test.js` |

## Runtime State: `.aiox/`

Alem dos diretorios de framework, o AIOX cria um diretorio de estado em runtime:

```
.aiox/
  project-status.yaml    # Status atual do projeto
  status.json            # Estado de runtime
  patterns/              # Padroes aprendidos (ADE)
  worktrees/             # Estado de worktrees (ADE)
  qa-loops/              # Estado de QA loops (ADE)
```

Esse diretorio e **gitignored** por padrao -- contem estado transiente que nao deve ir para o repositorio.

## Resumo

A estrutura do AIOX e organizada em quatro camadas:

1. **Framework** (`.aiox-core/`) -- O motor. Agentes, tasks, templates, quality gates
2. **Projeto** (`docs/`, `tests/`, `src/`) -- Seu codigo e documentacao
3. **IDE** (`.claude/`, `.cursor/`, `.gemini/`) -- Integracao com editores
4. **Runtime** (`.aiox/`) -- Estado transiente durante execucao

Essa separacao garante que o framework e portavel (move `.aiox-core/` para outro projeto e funciona), que seu codigo nao se mistura com o framework, e que cada IDE tem sua propria configuracao sem conflitos.
