# Estrutura do Projeto

Entender a estrutura de diretórios do AIOX é fundamental para trabalhar efetivamente com o framework. Cada pasta tem um propósito claro, e saber onde cada coisa vive evita confusão e acelera seu workflow.

---

## Visão Geral

O AIOX usa uma **arquitetura modular** com separação clara de responsabilidades:

1. **Framework Core** (`.aiox-core/`) -- Componentes portáveis organizados por domínio
2. **Project Workspace** (raiz) -- Implementação específica do projeto
3. **Documentation** (`docs/`) -- Documentação pública e stories

A filosofia é: **organização por domínio**, não por tipo de arquivo. Agentes ficam com agentes, templates ficam com templates, infraestrutura fica com infraestrutura.

## Módulos em Detalhe

### `.aiox-core/development/` -- Assets de Desenvolvimento

Este é o coração do framework. Aqui vivem os agentes, tasks e workflows.

**Agentes** (`.aiox-core/development/agents/`):

Cada agente é um arquivo Markdown com frontmatter YAML que define persona, comandos e dependências. São 11 agentes core:

| Arquivo | Agente | Papel |
|---|---|---|
| `aiox-master.md` | @aiox-master (Orion) | Orquestrador Mestre |
| `dev.md` | @dev (Dex) | Desenvolvedor Sênior |
| `qa.md` | @qa (Quinn) | Arquiteto de Testes |
| `architect.md` | @architect (Aria) | Arquiteta de Sistemas |
| `pm.md` | @pm (Morgan) | Product Manager |
| `po.md` | @po (Pax) | Product Owner |
| `sm.md` | @sm (River) | Scrum Master |
| `analyst.md` | @analyst (Atlas) | Analista de Negócios |
| `data-engineer.md` | @data-engineer (Dara) | Engenheira de Dados |
| `devops.md` | @devops (Gage) | Guardião de Repositórios |
| `ux-design-expert.md` | @ux-design-expert (Uma) | Designer UX/UI |

**Tasks** (`.aiox-core/development/tasks/`):

Mais de 60 tasks disponíveis. Cada task é um procedimento step-by-step que um agente executa. Exemplos:

- `create-next-story.md` -- Cria a próxima story do backlog
- `develop-story.md` -- Implementa uma story
- `spec-write-spec.md` -- Escreve especificação técnica

**Workflows** (`.aiox-core/development/workflows/`):

Workflows orquestram múltiplas tasks em sequência:

- `greenfield-fullstack.yaml` -- Projeto novo do zero
- `brownfield-service.yaml` -- Feature em projeto existente
- `spec-pipeline.yaml` -- Pipeline de especificação (ADE)
- `qa-loop.yaml` -- Loop de review e correção (ADE)

### `.aiox-core/core/` -- Motor do Framework

Componentes essenciais que fazem o framework funcionar:

| Diretório | Responsabilidade |
|---|---|
| `config/` | Sistema de configuração |
| `elicitation/` | Engine de prompting interativo |
| `quality-gates/` | Validadores de quality gates |
| `registry/` | Registro de serviços |
| `session/` | Gerenciamento de estado em runtime |
| `utils/` | Utilitários compartilhados |

<Callout type="info" title="Você raramente edita core/">
O diretório core/ é gerenciado pelo framework. Você interage com ele indiretamente através de agentes e configuração. A menos que esteja contribuindo para o AIOX, não modifique esses arquivos.
</Callout>

### `.aiox-core/data/` -- Dados Compartilhados

Arquivos de dados que agentes leem durante a ativação:

| Arquivo | Dono | Função |
|---|---|---|
| `aiox-kb.md` | @aiox-master | Knowledge base do AIOX (lazy-loaded) |
| `agent-config-requirements.yaml` | @architect | Regras de carregamento por agente |
| `technical-preferences.md` | @architect | Preferências técnicas do time |
| `workflow-patterns.yaml` | @sm | Padrões de detecção de workflow |

Cada arquivo tem um **dono** (agente responsável) e um **trigger de atualização** definido. Isso garante que dados compartilhados não fiquem desatualizados.

## Convenções de Nomenclatura

O AIOX segue regras estritas de nomenclatura:

```yaml
Diretorios: kebab-case
  .aiox-core/, development/, quality-gates/

Arquivos de codigo: kebab-case + extensao
  agent-executor.js, task-runner.js

Arquivos de docs: kebab-case + .md
  coding-standards.md, story-6.1.2.5.md

Templates: {nome}-tmpl.{yaml|md}
  story-tmpl.yaml, prd-tmpl.md

Checklists: {nome}-checklist.md
  architect-checklist.md

Stories: story-{epic}.{story}.{substory}.md
  story-6.1.2.5.md

Epics: epic-{numero}-{nome}.md
  epic-6.1-agent-identity-system.md
```

<Callout type="warning" title="Siga as convenções">
Arquivos fora do padrão podem não ser detectados pelo framework. Se você criar um agente chamado `MyAgent.MD` em vez de `my-agent.md`, ele pode não ser carregado corretamente.
</Callout>

## Onde Colocar Novos Arquivos

Uma referência rápida para quando você precisar criar algo:

| Você quer criar... | Coloque em... |
|---|---|
| Novo agente | `.aiox-core/development/agents/{nome}.md` |
| Nova task | `.aiox-core/development/tasks/{nome}.md` |
| Novo workflow | `.aiox-core/development/workflows/{nome}.yaml` |
| Novo template | `.aiox-core/product/templates/{nome}-tmpl.yaml` |
| Nova checklist | `.aiox-core/product/checklists/{nome}-checklist.md` |
| Novo CLI command | `.aiox-core/cli/commands/{categoria}/{nome}.js` |
| Nova story | `docs/stories/{contexto}/{nome}.md` |
| Novo teste | `tests/{tipo}/{nome}.test.js` |

## Runtime State: `.aiox/`

Além dos diretórios de framework, o AIOX cria um diretório de estado em runtime:

```
.aiox/
  project-status.yaml    # Status atual do projeto
  status.json            # Estado de runtime
  patterns/              # Padroes aprendidos (ADE)
  worktrees/             # Estado de worktrees (ADE)
  qa-loops/              # Estado de QA loops (ADE)
```

Esse diretório é **gitignored** por padrão -- contém estado transiente que não deve ir para o repositório.

---

## Resumo

A estrutura do AIOX é organizada em quatro camadas:

1. **Framework** (`.aiox-core/`) -- O motor. Agentes, tasks, templates, quality gates
2. **Projeto** (`docs/`, `tests/`, `src/`) -- Seu código e documentação
3. **IDE** (`.claude/`, `.cursor/`, `.gemini/`) -- Integração com editores
4. **Runtime** (`.aiox/`) -- Estado transiente durante execução

Essa separação garante que o framework é portável (move `.aiox-core/` para outro projeto e funciona), que seu código não se mistura com o framework, e que cada IDE tem sua própria configuração sem conflitos.
