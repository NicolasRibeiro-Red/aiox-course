# Primeiro Valor em 10 Minutos

O AIOX tem uma regra interna: **primeiro valor em 10 minutos ou menos**. Isso significa que, a partir de um projeto instalado, você deve conseguir ativar um agente, interagir com ele e receber output útil em no máximo 10 minutos. Vamos fazer isso agora.

---

## O que é "Primeiro Valor"?

O AIOX define primeiro valor como três condições cumpridas:

1. Você **ativa** um agente AIOX
2. Você recebe uma **resposta válida** de ativação (greeting)
3. Você roda um **comando starter** (`*help`) e recebe output útil

Se as três condições forem atendidas em até 10 minutos, o setup está correto.

<Callout type="info" title="Por que 10 minutos?">
Essa regra existe para detectar problemas de instalação cedo. Se você não consegue atingir primeiro valor em 10 minutos, algo na configuração está errado -- e é mais fácil diagnosticar agora do que depois de horas tentando usar o framework.
</Callout>

---

## Passo 1: Certifique-se de que o AIOX está Instalado

Se você seguiu as lições anteriores, seu projeto já está pronto. Verifique:

```bash
cd my-first-project
npx aiox-core doctor
```

O doctor deve reportar todos os checks como OK. Se algo falhar, volte para as lições de instalação e plataforma.

---

## Passo 2: Abra seu IDE

O próximo passo depende do IDE que você está usando.

### Claude Code

```bash
# Se ainda nao instalou globalmente
npm install -g @anthropic-ai/claude-code

# Abrir Claude Code no diretorio do projeto
cd my-first-project
claude
```

### Gemini CLI

```bash
# Abrir Gemini no diretorio do projeto
cd my-first-project
gemini
```

### Cursor

Abra o diretório do projeto no Cursor normalmente. As regras AIOX foram instaladas em `.cursor/rules/`.

---

## Passo 3: Ativar seu Primeiro Agente

Agora vem a parte boa. Vamos ativar o **@aiox-master** (Orion) -- o orquestrador mestre que conhece todos os outros agentes.

| IDE | Comando de Ativação |
|---|---|
| Claude Code | `/aiox-master` |
| Gemini CLI | `/aiox-menu` depois `/aiox-aiox-master` |
| Codex CLI | `/skills` depois `aiox-aiox-master` |
| Cursor | `@aiox-master` no chat |

Após a ativação, você deve receber um **greeting** -- uma mensagem de apresentação do agente com seu nome, papel e capabilities. Algo como:

```
Orion (AIOX Master) ativado.
Orquestrador Mestre do framework AIOX.
Digite *help para ver comandos disponiveis.
```

<Callout type="tip" title="Se não receber greeting">
1. Verifique se os arquivos de agente existem em `.aiox-core/development/agents/`
2. Rode `npm run sync:ide` para resincronizar
3. Reinicie o IDE/CLI
4. Rode `npx aiox-core doctor` para diagnóstico
</Callout>

---

## Passo 4: Rodar *help

Com o agente ativo, digite:

```
*help
```

O comando `*help` é **universal** -- funciona em qualquer agente AIOX. Ele retorna:

- Lista de comandos disponíveis para aquele agente
- Breve descrição de cada comando
- Exemplos de uso

## Passo 5: Explorar Outros Agentes

Agora que você sabe o básico, vamos experimentar com mais agentes. Saia do aiox-master e ative outro:

```
*exit
```

Agora ative o agente `@dev` (Dex) -- o desenvolvedor sênior:

| IDE | Comando |
|---|---|
| Claude Code | `/dev` |
| Gemini CLI | `/aiox-dev` |
| Cursor | `@dev` |

Rode `*help` novamente e note como os comandos são diferentes -- cada agente tem seu próprio set de comandos especializados.

### Exemplos de Interação com Agentes

**Com @dev (Dex):**

```
*help
# Mostra comandos de desenvolvimento: implementacao, refactoring, debug
```

**Com @pm (Morgan):**

```
*help
# Mostra comandos de produto: criar PRD, criar epic, roadmap
```

**Com @qa (Quinn):**

```
*help
# Mostra comandos de qualidade: review, testes, audit
```

## O Fluxo Típico de Trabalho

Agora que você ativou agentes, vale entender o fluxo completo que você vai usar no dia a dia:

```
1. @pm cria um PRD              *create-epic
2. @sm faz o draft das stories  *draft
3. @po valida as stories        *validate-story-draft
4. @dev implementa              (trabalha a partir do arquivo de story)
5. @qa revisa                   *review STORY-ID
6. @devops faz push             *push
7. @po fecha a story            *close-story STORY-ID
```

<Callout type="warning" title="Não pule etapas">
Esse fluxo existe por um motivo. Cada etapa tem um quality gate que valida o trabalho antes de avançar. Pular de @pm direto para @dev funciona tecnicamente, mas você perde a validação de arquitetura e stories -- e isso cobra depois.
</Callout>

## Brownfield: Primeiro Uso em Projeto Existente

Se você instalou o AIOX em um projeto existente, o primeiro uso é ligeiramente diferente:

| Situação | Workflow Recomendado |
|---|---|
| Adicionar feature grande | `@pm` seguido de `*create-doc brownfield-prd` |
| Auditar codebase legado | Workflow `brownfield-discovery.yaml` |
| Enhancement rápido | `@pm` seguido de `*brownfield-create-epic` |
| Bug fix isolado | `@pm` seguido de `*brownfield-create-story` |

Na primeira ativação em brownfield, o agente oferece analisar o codebase:

```
Detectei codigo existente sem documentacao AIOX.
Posso analisar o codebase e gerar:
- Documentacao de arquitetura
- Relatorio de debito tecnico

Isso leva 4-8 horas. Deseja continuar?
```

Aceitar essa análise é opcional mas altamente recomendado -- ela cria a base de contexto que todos os agentes vão usar.

## Checklist de Primeiro Valor

Confirme que você completou tudo:

- [ ] AIOX instalado e `doctor` sem erros
- [ ] IDE aberto no diretório do projeto
- [ ] Agente ativado com greeting recebido
- [ ] `*help` executado com output válido
- [ ] Pelo menos 2 agentes testados
- [ ] Entendeu o fluxo típico de trabalho

---

## Diagnóstico: Se Algo Deu Errado

| Sintoma | Causa Provável | Solução |
|---|---|---|
| Agente não responde | IDE não suportado ou arquivos faltando | `npm run sync:ide` + reiniciar IDE |
| Greeting incompleto | Arquivo de agente corrompido | Reinstalar: `npx aiox-core install` |
| `*help` não reconhecido | Agente não ativado corretamente | Verifique o comando de ativação do seu IDE |
| Comandos retornam erro | Dependências de task faltando | `npx aiox-core doctor --fix` |
